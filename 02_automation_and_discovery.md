# 第 2 章 · 自动化与发现 —— DHCP、ARP、DNS

> **本章目标：** 搞懂三个"自动化"协议——设备怎么自动拿到 IP（DHCP）、怎么通过 IP 找到对方的网卡（ARP）、怎么把域名翻译成 IP（DNS）。这三个是网络世界的"基础设施"，出了问题整个网络就瘫了。

---

## 🚦 新手入口（先跑通逻辑）

### 30 秒先懂

- `DHCP` 解决"我是谁（我的 IP 是多少）"
- `ARP` 解决"同网段里你在哪（你的 MAC 是什么）"
- `DNS` 解决"目标叫什么（域名对应哪个 IP）"

### 最小可用（先会排查）

1. 设备拿不到 IP，先看 DHCP。
2. 同网段互访异常，先看 ARP。
3. IP 能通但域名不通，先看 DNS。

### 阅读顺序（建议）

- 按 `DHCP → ARP → DNS` 看，顺序就是一次联网的启动顺序
- hosts 文件放最后看，当作本地覆盖技巧

## 2.1 DHCP —— 自动分配 IP

### 一句话概括

🔑 **DHCP 就是"自动给设备分配 IP 地址"的服务。** 没有它，你每台设备都得手动配 IP。

### 生活类比

你去酒店住宿：
- **没有 DHCP**：你得自己选房间号，还得确保没人住 → 手动配 IP
- **有 DHCP**：前台自动给你分一个空房间，退房后房间回收 → 自动分配

### DHCP 工作过程（四步）

```
你的设备（刚连上 Wi-Fi）                    路由器（DHCP 服务器）
    │                                           │
    │  ① DISCOVER: "有人能给我分个IP吗？"        │
    │ ─────────────────────────────────────────→ │
    │                                           │
    │  ② OFFER: "192.168.1.100 给你用吧"         │
    │ ←───────────────────────────────────────── │
    │                                           │
    │  ③ REQUEST: "好的我要这个IP"                │
    │ ─────────────────────────────────────────→ │
    │                                           │
    │  ④ ACK: "确认，这个IP是你的了，租期24小时"   │
    │ ←───────────────────────────────────────── │
```

> 💡 **四步的英文名字其实很好记**：DORA（Discover → Offer → Request → Acknowledge）
> - **Discover**（发现）：设备广播"谁能给我 IP？"
> - **Offer**（提供）：DHCP 服务器回复"我能给你这个 IP"
> - **Request**（请求）：设备确认"我要这个"
> - **Acknowledge**（确认）：服务器最终确认"成交"

### DHCP 返回了什么？

DHCP 不光给你一个 IP，还会顺便告诉你一堆网络配置：

| 信息 | 例子 | 作用 |
|-----|------|------|
| **IP 地址** | `192.168.1.100` | 你的"门牌号" |
| **子网掩码** | `255.255.255.0` | 判断谁在同一个局域网 |
| **默认网关** | `192.168.1.1` | 出去上网要走的"门" |
| **DNS 服务器** | `192.168.1.1` 或 `8.8.8.8` | 域名翻译服务 |
| **租期** | 24小时 | 这个 IP 能用多久 |
| **域名** | `home`（可选） | 本地域名后缀 |

所以你连上WiFi 后，这些配置**全自动**就有了，不用手动填。

> 🤔 **DNS 为什么可能是路由器的 IP？**
>
> 路由器可以充当"DNS 转发器"——你的设备问路由器，路由器再帮你去问真正的 DNS（比如 8.8.8.8）。好处是路由器可以缓存结果，下次更快。
>
> 你也可以在设备上手动改成 `8.8.8.8`，这样就跳过路由器，直接问 Google。

### 关键概念

| 概念 | 解释 |
|---|---|
| **地址池** | 路由器可以分配的 IP 范围，比如 192.168.1.100 ~ 192.168.1.200 |
| **租约（Lease）** | IP 不是永久的，有"过期时间"，到期要续租或回收 |
| **静态绑定** | 把某个 MAC 地址固定绑定到某个 IP，比如让 NAS 永远是 192.168.1.50 |
| **DHCP 服务器** | 通常就是你的路由器，企业里可能是专门的服务器 |

### ⚠️ 常见坑

**坑 1：地址池耗尽**
- 现象：新设备连上 Wi-Fi 但拿不到 IP
- 原因：地址池里的 IP 都被占了（设备太多或租约太长）
- 解决：扩大地址池范围，或缩短租约时间

**坑 2：IP 冲突**
- 现象：两台设备拿到同一个 IP，一台或两台都上不了网
- 原因：有人手动配了一个 DHCP 地址池范围内的 IP
- 解决：手动配的 IP 要在地址池范围**之外**

**坑 3：169.254.x.x 地址**
- 现象：设备 IP 变成 169.254 开头
- 原因：DHCP 失败了！设备找不到 DHCP 服务器，自己随便分了一个
- 🔑 **看到 169.254 就知道是 DHCP 出了问题**

### 💻 动手验证

```bash
# Mac —— 查看当前 DHCP 分配的信息
ipconfig getpacket en0

# Linux —— 查看 DHCP 租约
cat /var/lib/dhcp/dhclient.leases
# 或
journalctl -u NetworkManager | grep DHCP

# 通用 —— 查看当前 IP（看是不是 DHCP 分配的）
# Mac
ifconfig en0
# Linux
ip addr show
```

## 2.2 ARP（Address Resolution Protocol，地址解析协议）—— 通过 IP 找 MAC 地址

### 一句话概括

🔑 **ARP 就是"我知道你的 IP，但我需要知道你的 MAC 地址才能发数据给你"。**

### 为什么需要 ARP？

在局域网内，数据最终是通过 **MAC 地址**（网卡的物理地址）来投递的，不是 IP。

就像你知道同事的工号（IP），但要把文件递给他，你得知道他坐在哪个工位（MAC）。

#### 为什么局域网必须用 MAC？

因为局域网底层是**以太网**，以太网的数据帧只认 MAC 地址：

```
以太网数据帧结构：
┌──────────────┬──────────────┬─────────┐
│ 目标 MAC     │ 源 MAC       │ 数据    │
│ AA:BB:...    │ 11:22:...    │ ...     │
└──────────────┴──────────────┴─────────┘
       ↑
   网卡只看这个，不看 IP
```

所以即使你知道目标 IP，没有 MAC 也发不出去。

#### 那广域网呢？也需要 ARP 吗？

**不需要**。广域网不用以太网，也就不用 MAC 和 ARP。

数据包在互联网上传输，是**一跳一跳**经过多个路由器：

```
你的电脑 → 路由器A → 路由器B → 路由器C → 目标服务器
   ↑          ↑          ↑          ↑
  以太网    以太网     光纤/MPLS   以太网
  用ARP     用ARP     不用ARP     用ARP
  用MAC     用MAC     用其他地址   用MAC
```

每一跳用什么协议，就用什么地址：
- 以太网段：用 MAC 地址，需要 ARP
- 光纤/MPLS段：用标签或其他方式，不用 ARP

#### IP 地址 vs 链路层地址

| | IP 地址 | 链路层地址（如 MAC） |
|--|--------|-------------------|
| 范围 | **端到端**，全程不变 | **逐跳**，每段都换 |
| 作用 | 找最终目的地 | 在当前链路上传输 |

```
例子：从你家访问百度

IP 地址：192.168.1.100 → 39.156.66.10（全程不变，贴在数据包上）

链路层地址（每段不一样）：
  你家 → 路由器：    MAC_A → MAC_B（用以太网）
  路由器 → 运营商：  MAC_C → MAC_D（用以太网）
  运营商 → 骨干网：  MPLS标签（不用MAC）
  ...
```

**类比**：寄快递从北京到广州

| | 网络 | 类比 |
|--|------|------|
| IP 地址 | 收件人地址 | 广州市xx区xx号（写在包裹上，全程不变） |
| 链路层地址 | 每一程的运输 | 北京→郑州用火车，郑州→武汉用飞机（每段不同） |

### 生活类比

你在办公室里喊："工号 1100 的同事是谁？请举手！"
- 所有人都听到了（**广播**）
- 工号 1100 的人举手回应："是我，我坐在 A3 工位"
- 你记下来：工号 1100 = A3 工位（**ARP 缓存**）
- 下次找他就直接去 A3，不用再喊了

### ARP 工作过程

```
你的 Mac（要发数据给 192.168.1.50）
    │
    │  ① ARP 请求（广播）："谁是 192.168.1.50？告诉我你的 MAC 地址"
    │ ─────────────→ 局域网内所有设备都收到
    │
    │  ② ARP 响应（单播）："我是 192.168.1.50，MAC 是 AA:BB:CC:DD:EE:FF"
    │ ←───────────── 只有目标设备回复
    │
    │  ③ 记入 ARP 表，下次直接用，不再广播
```

### ⚠️ ARP 欺骗（了解即可）

ARP 协议**没有验证机制**——任何人都可以说"我是 192.168.1.1"。

攻击者可以冒充网关，让你的数据先经过他再转发，从而**窃听或篡改**你的流量。这就是"中间人攻击"的一种。

> 防范：企业网络用 DAI（Dynamic ARP Inspection，动态 ARP 检测）；个人用 HTTPS 加密就够了。

### 💻 动手验证

```bash
# Mac / Linux —— 查看 ARP 缓存表
arp -a

# 输出示例：
# ? (192.168.1.1) at aa:bb:cc:dd:ee:ff on en0 [ethernet]
# ? (192.168.1.50) at 11:22:33:44:55:66 on en0 [ethernet]
#
# 解读：192.168.1.1（路由器）的 MAC 地址是 aa:bb:cc:dd:ee:ff
```

## 2.3 DNS（Domain Name System，域名系统）—— 域名翻译成 IP

### 一句话概括

🔑 **DNS 就是互联网的"电话簿"，把人类能记住的域名（baidu.com）翻译成机器需要的 IP 地址（39.156.66.10）。**

### 为什么需要 DNS？

你能记住 `baidu.com`，但你记不住 `39.156.66.10`。DNS 帮你做翻译。

### DNS 查询过程

当你输入 `www.baidu.com`，查询过程是这样的：

```
① 浏览器缓存 → 查过就直接用
    ↓ 没有
② 操作系统缓存 → 系统级缓存
    ↓ 没有
③ hosts 文件 → 本地手动配置的映射
    ↓ 没有
④ 本地 DNS 服务器（通常是路由器或运营商 DNS）
    ↓ 没有
⑤ 根 DNS 服务器 → "我不知道，但 .com 的管理者是这个"
    ↓
⑥ .com 顶级域服务器 → "baidu.com 的管理者是这个"
    ↓
⑦ baidu.com 权威 DNS → "www.baidu.com 的 IP 是 39.156.66.10"
```

🔑 **日常你只需要知道：先查缓存 → 再查 hosts → 最后问 DNS 服务器。**

### DNS 记录类型

| 类型 | 作用 | 例子 |
|---|---|---|
| **A** | 域名 → IPv4 地址 | `baidu.com → 39.156.66.10` |
| **AAAA** | 域名 → IPv6 地址 | `baidu.com → 2400:da00::dbf:...` |
| **CNAME**（Canonical Name，别名记录） | 域名 → 另一个域名（别名） | `www.baidu.com → www.a.shifen.com` |
| **MX**（Mail Exchange，邮件交换） | 邮件服务器 | `baidu.com → mx.baidu.com` |
| **TXT** | 文本信息（验证用） | 域名所有权验证、SPF（Sender Policy Framework，发件人策略框架）邮件防伪 |
| **NS**（Name Server，域名服务器） | 域名的 DNS 服务器 | `baidu.com → ns1.baidu.com` |

🔑 **日常最常用的就是 A 记录和 CNAME 记录。**

> 🤔 **CNAME 有什么用？为什么不直接用 A 记录？**
> 
> 假设你有 `www.example.com`、`blog.example.com`、`shop.example.com` 三个子域名，都指向同一台服务器（IP: 1.2.3.4）。如果用 A 记录，你要写三条记录。有一天服务器换了 IP，你要改三个地方。
> 
> 用 CNAME 就方便了：让 `www`、`blog`、`shop` 都 CNAME 到 `example.com`，然后只给 `example.com` 写一条 A 记录。换 IP 时只改一个地方就行。
> 
> **特别是用了 CDN 的情况**：CDN 服务商给你一个域名（如 `xxx.cdn.com`），你用 CNAME 指过去，CDN 会自动解析到离用户最近的服务器 IP。你根本不需要知道具体 IP 是什么。

### TTL 和缓存

每条 DNS 记录都有一个 **TTL（Time To Live，生存时间）**，表示"这条记录可以缓存多久"。

- TTL = 300 → 缓存 5 分钟，5 分钟后重新查询
- TTL = 86400 → 缓存 1 天

🔑 **改了 DNS 记录后不是立刻生效的**，要等旧缓存过期。这就是为什么"DNS 修改需要等待生效"。

> 🤔 **TTL 到期后会怎样？**
> 
> TTL 到期后，缓存自动失效。下次有人查询这个域名时，会重新去 DNS 服务器获取最新的 IP 地址。
> 
> **实际影响**：如果你改了 DNS 记录（比如把服务器从 A 搬到 B），旧的 TTL 没过期之前，部分用户还是会访问到旧服务器。所以搬迁前建议先把 TTL 改短（比如 60 秒），等旧缓存过期后再改 IP，这样切换更快。

### 常用公共 DNS

| DNS 服务器 | IP | 特点 |
|---|---|---|
| Google DNS | `8.8.8.8` / `8.8.4.4` | 全球通用，国内可能慢 |
| Cloudflare DNS | `1.1.1.1` / `1.0.0.1` | 注重隐私，速度快 |
| 阿里 DNS | `223.5.5.5` / `223.6.6.6` | 国内推荐 |
| 114 DNS | `114.114.114.114` | 国内老牌 |

⚠️ **运营商 DNS 可能会"劫持"**——你访问一个不存在的域名，它不返回错误，而是跳转到广告页。换成公共 DNS 可以解决。

### 自建局域网 DNS：原理和做法

你可能会问：公网 DNS（如 8.8.8.8）知道 baidu.com 的 IP，是因为百度在域名注册商那里登记了。但如果我自己搞一个局域网 DNS，它怎么知道存什么呢？

#### DNS 服务器的本质

**DNS 服务器就是一个"查表程序"**：

```
1. 监听 53 端口，等待查询请求
2. 收到 "xxx是什么IP？" 的请求
3. 查自己的记录表
4. 有记录 → 返回 IP
   没记录 → 转发给上游 DNS
```

#### 自建 DNS 的工作流程

就是两个功能：**本地记录 + 转发上游**

```
收到查询 "nas.home.local 是什么？"
    ↓
查本地记录表 → 找到了！→ 返回 192.168.1.50


收到查询 "baidu.com 是什么？"
    ↓
查本地记录表 → 没找到
    ↓
转发给上游 DNS（8.8.8.8）→ 返回 39.156.66.10
    ↓
缓存起来，下次直接回答
```

#### 怎么配置？

以 **dnsmasq** 为例（最轻量的 DNS 软件）：

**1. 安装**

```bash
# Mac
brew install dnsmasq

# Ubuntu/Debian
sudo apt install dnsmasq
```

**2. 配置文件**

```
# /usr/local/etc/dnsmasq.conf (Mac)
# 或 /etc/dnsmasq.conf (Linux)

# ====== 转发配置：本地查不到时，问谁？======
server=8.8.8.8          # 上游 DNS 1
server=223.5.5.5        # 上游 DNS 2（备用）

# ====== 本地记录：你自己定义的域名 ======
address=/nas.home.local/192.168.1.50
address=/git.home.local/192.168.1.51
address=/printer.home.local/192.168.1.52
```

**配置解释**：

| 配置项 | 含义 |
|-------|------|
| `server=8.8.8.8` | 本地查不到时，转发给 8.8.8.8 帮你查 |
| `address=/nas.home.local/192.168.1.50` | 有人问 nas.home.local，直接返回 192.168.1.50 |

**3. 启动服务**

```bash
# Mac
sudo brew services start dnsmasq

# Linux
sudo systemctl start dnsmasq
```

**4. 让局域网设备用这个 DNS**

在路由器的 DHCP 设置里，把 DNS 服务器地址改成你自建 DNS 的 IP（比如 `192.168.1.100`）。

这样所有设备连上 WiFi 后，DHCP 会自动告诉它们"DNS 用 192.168.1.100"。

**5. 验证**

```bash
# 查本地记录
nslookup nas.home.local 192.168.1.100
# 返回: 192.168.1.50

# 查公网域名（走转发）
nslookup baidu.com 192.168.1.100
# 返回: 39.156.66.10（第一次会慢一点，第二次就走缓存了）
```

#### 一张图总结

```
┌─────────────────────────────────────────────────────┐
│            自建 DNS 服务器 (192.168.1.100)          │
│                                                     │
│  本地记录表（你配的）：                              │
│    nas.home.local   → 192.168.1.50                 │
│    git.home.local   → 192.168.1.51                 │
│                                                     │
│  上游 DNS（转发用的）：                              │
│    server=8.8.8.8                                  │
│    server=223.5.5.5                                │
│                                                     │
│  工作流程：                                         │
│    收到查询 → 查本地表 → 有？返回                   │
│                       → 没有？转发给 8.8.8.8        │
└─────────────────────────────────────────────────────┘
                         ↑
        局域网设备 DNS 都设成 192.168.1.100
```

> 💡 **更简单的方案**：如果只是想让内网设备能用域名互相访问，用 mDNS（Multicast DNS）更省事，不用搭 DNS 服务器。详见本章后面的 mDNS 章节。

### 💻 动手验证

```bash
# nslookup —— 最简单的 DNS 查询（Mac / Linux）
nslookup baidu.com
# 输出：
# Server:    192.168.1.1        ← 你用的 DNS 服务器
# Address:   39.156.66.10       ← 查询结果

# 指定用 Google DNS 查询
nslookup baidu.com 8.8.8.8

# dig —— 更详细的 DNS 查询（推荐）
dig baidu.com
# 看 ANSWER SECTION 部分：
# baidu.com.    300    IN    A    39.156.66.10
#               ↑TTL         ↑类型  ↑IP地址

# 只看精简结果
dig +short baidu.com

# 查看 CNAME 记录
dig www.baidu.com CNAME

# 清除本地 DNS 缓存
# Mac
sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder
# Linux (systemd-resolved)
sudo systemd-resolve --flush-caches
```

## 2.4 hosts 文件 —— 本地 DNS 覆盖

### 一句话概括

🔑 **hosts 文件是本地的"小电话簿"，优先级比 DNS 服务器高。**

### 文件位置

```bash
# Mac / Linux
/etc/hosts

# Windows
C:\Windows\System32\drivers\etc\hosts
```

### 文件格式

```
# IP地址        域名
127.0.0.1       localhost
192.168.1.100   mynas.home
93.184.216.34   example.com
```

### 常见用途

| 用途 | 例子 |
|---|---|
| 本地开发 | `127.0.0.1 myapp.local` → 让 myapp.local 指向本机 |
| 屏蔽网站 | `0.0.0.0 ads.example.com` → 让广告域名指向空地址 |
| 测试 | 把域名临时指向测试服务器 IP |
| 绕过 DNS 污染 | 手动写入正确的 IP |

### 💻 动手操作

```bash
# 查看 hosts 文件
cat /etc/hosts

# 编辑 hosts 文件（需要 sudo）
sudo nano /etc/hosts
# 或
sudo vim /etc/hosts

# 添加一行后保存，立刻生效（不需要重启）
# 验证：
ping mynas.home
```

⚠️ **hosts 文件优先级高于 DNS**，如果你在 hosts 里写了错误的映射，会导致域名访问异常。排查问题时记得检查 hosts 文件。

## 2.5 三者关系总结

```
设备刚连上网络时：
  DHCP → 自动获取 IP 地址、子网掩码、网关、DNS 服务器

设备要和局域网内的其他设备通信时：
  ARP → 通过 IP 找到对方的 MAC 地址

设备要访问互联网上的域名时：
  DNS → 把域名翻译成 IP 地址
```

**打个比方：**
- **DHCP** = 入职时 HR 给你分工号和工位 → 分配 IP
- **ARP** = 你喊"工号 1100 坐哪？" → 通过 IP 找 MAC
- **DNS** = 你查通讯录"张三的工号是多少？" → 通过域名找 IP

🔑 **三者的执行顺序：** 先 DHCP 拿到 IP → 通信时用 ARP 找 MAC → 访问域名时用 DNS 查 IP

## 本章小结

| 协议 | 解决什么问题 | 类比 |
|---|---|---|
| DHCP | 自动分配 IP 地址 | 酒店前台分房间 |
| ARP | 通过 IP 找 MAC 地址 | 办公室喊人找工位 |
| DNS | 域名翻译成 IP | 电话簿查号码 |
| hosts | 本地 DNS 覆盖 | 自己的小本本 |

**🔑 排查口诀：**
- 拿不到 IP → 查 DHCP
- 局域网内 ping 不通 → 查 ARP 表
- 域名打不开但 IP 能访问 → 查 DNS
- DNS 正常但域名还是不对 → 查 hosts 文件

## 📦 扩展区域

> 以下内容日常不常用，感兴趣可以了解：

### DHCP 中继（DHCP Relay）

大型网络中，DHCP 服务器和客户端可能不在同一个网段。这时需要在路由器上配置 DHCP Relay，把 DHCP 请求转发到另一个网段的 DHCP 服务器。

### DNS over HTTPS（DoH）/ DNS over TLS（DoT）—— 加密的 DNS 查询

传统 DNS 查询是明文的，运营商可以看到你查了什么域名。DoH/DoT 对 DNS 查询进行加密：
- **DoH**：通过 HTTPS 发送 DNS 查询（端口 443）
- **DoT**：通过 TLS 发送 DNS 查询（端口 853）

Firefox、Chrome 都支持 DoH，可以在设置里开启。

### mDNS（Multicast DNS，多播 DNS）

局域网内的设备发现协议。Mac 上的 `.local` 域名就是用 mDNS 解析的：
```bash
# 比如你的 Mac 叫 "my-mac"，局域网内其他设备可以用：
ping my-mac.local
```
