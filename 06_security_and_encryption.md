## 这章解决什么问题？

你有没有想过：
- 在咖啡厅连公共 Wi-Fi 登录网站，密码会不会被偷看？

## 1. 为什么需要加密？

### 🔑 HTTP 是明文传输的

想象一下，你在咖啡厅连上公共 Wi-Fi，打开一个 `http://` 开头的网站，输入账号密码登录。这时候，你的数据是怎么传输的？

**答案：像明信片一样，所有经手的人都能看到内容。**

HTTP 协议本身不加密，数据在网络上传输时是明文的。从你的电脑到服务器，中间要经过：
- 咖啡厅的路由器
- 运营商的交换机
- 各种中转路由器

这些设备上的任何人，只要愿意，都能截获你的数据包，看到你输入的账号密码。

### 生活类比

- **HTTP**：像寄明信片，邮递员、分拣员都能看到内容
- **HTTPS**：像寄密封信，只有收件人能打开

### 实际风险

⚠️ **真实案例**：
- 公共 Wi-Fi 下用 HTTP 登录网站 → 密码可能被截获
- 中间人攻击（Man-in-the-Middle）：黑客伪装成路由器，截获所有流量
- 运营商劫持：有些运营商会在 HTTP 页面里插入广告（因为能看到内容）

**所以现在的网站基本都用 HTTPS 了。**

## 2. 对称加密 vs 非对称加密

加密的核心问题：**怎么让只有你和对方能看懂消息，中间人看不懂？**

### 🔑 对称加密：一把钥匙走天下

**一句话概括**：加密和解密用同一把钥匙。

**生活类比**：
你和朋友约定一个暗号："把每个字母往后移 3 位"。
- 你发消息：`HELLO` → 加密成 `KHOOR`
- 朋友收到：`KHOOR` → 解密成 `HELLO`

双方都知道这个规则（钥匙），所以能互相看懂。

**特点**：
- ✅ 速度快（计算简单）
- ❌ 问题：钥匙怎么安全地传给对方？

如果你在网上把"往后移 3 位"这个规则告诉朋友，中间人也知道了，加密就没用了。

### 🔑 非对称加密：两把钥匙的魔法

**一句话概括**：有两把钥匙（公钥 + 私钥），公钥加密的只有私钥能解，私钥加密的只有公钥能解。

**生活类比**：
想象一个特殊的邮箱：
- **公钥**：邮箱的投信口，任何人都能往里塞信
- **私钥**：邮箱的钥匙，只有你有，只有你能打开

别人用你的公钥加密消息 → 只有你的私钥能解密 → 中间人截获了也没用。

**解决了什么问题**：
- 公钥可以公开发布（比如放在网站上）
- 私钥自己保管，不需要在网络上传输
- 不用担心"钥匙怎么传"的问题

**实际使用：混合加密**

⚠️ **重点**：真实的 HTTPS 不是只用非对称加密，而是：
1. **先用非对称加密交换密钥**（慢但安全）
2. **再用对称加密传数据**（快）

为什么？因为非对称加密计算量大，速度慢。所以：
- 握手阶段：用非对称加密协商一个"临时密钥"
- 通信阶段：用这个临时密钥做对称加密

**兼顾了安全和速度。**

### ⚠️ 不需要记住具体算法

你可能听过 AES、RSA、DES 这些名字，**不用记**。知道原理就行：
- 对称加密：一把钥匙
- 非对称加密：两把钥匙

## 3. TLS/SSL — HTTPS 背后的加密机制

### 🔑 HTTPS = HTTP + TLS

- **SSL**（Secure Sockets Layer）：老版本的加密协议
- **TLS**（Transport Layer Security）：SSL 的升级版
- 现在说的"SSL"其实都是指 TLS（只是习惯叫法）

**HTTPS 就是在 HTTP 外面套了一层 TLS 加密。**

### TLS 握手过程（极简版）

当你访问 `https://www.example.com` 时，背后发生了什么？

```
1. 浏览器 → 服务器：你好，我支持这些加密方式（列表）
2. 服务器 → 浏览器：我选这个加密方式，这是我的证书（包含公钥）
3. 浏览器：验证证书没问题，生成一个临时密钥，用服务器公钥加密发过去
4. 服务器：用私钥解密，拿到临时密钥
5. 双方：开始用临时密钥做对称加密通信
```

**整个过程不到 1 秒，你感觉不到。**

### 🔗 概念关联

- 第 3 章讲的 HTTPS 端口 443，就是 TLS 加密的 HTTP
- SSH 也用类似的加密机制（后面会讲）

## 4. 证书（Certificate）：网站的"身份证"

### 🔑 证书是什么？

**一句话概括**：证明"这个网站确实是它声称的那个"。

**为什么需要证书？**

假设没有证书，只有加密：
- 你访问 `https://bank.com`
- 黑客伪装成 `bank.com`，给你发他的公钥
- 你用黑客的公钥加密密码 → 黑客用他的私钥解密 → 密码被盗

**证书就是用来防止这种"冒充"的。**

### 证书里包含什么？

打开浏览器，访问任意 HTTPS 网站，点地址栏的锁图标 → 查看证书，你会看到：

```
- 颁发给：www.example.com（域名）
- 颁发者：Let's Encrypt（CA 机构）
- 有效期：2025-01-01 到 2026-01-01
- 公钥：一串很长的数字
- 签名：CA 用自己的私钥签的名
```

### 🔑 CA（证书颁发机构）：发身份证的"公安局"

**CA 是谁？**
- Certificate Authority，证书颁发机构
- 就像公安局发身份证，CA 发网站证书

**CA 怎么工作？**
1. 网站向 CA 申请证书："我是 example.com，这是我的公钥"
2. CA 验证你确实拥有这个域名（比如让你在域名下放个特定文件）
3. CA 用自己的私钥给你的证书签名 → 颁发证书
4. 浏览器内置了一批信任的 CA（根证书）
5. 浏览器验证证书时，检查签名是不是信任的 CA 签的

**生活类比**：
- 你拿身份证去银行开户，银行信任公安局发的身份证
- 浏览器拿到网站证书，信任 CA 签发的证书

### 证书链：层层信任

实际上，证书是一条链：

```
根 CA（浏览器内置）
  ↓ 签名
中间 CA
  ↓ 签名
网站证书（example.com）
```

浏览器验证时，会一层层往上查，直到找到内置的根 CA。

### 🔑 Let's Encrypt：免费的 CA

以前申请证书要花钱（几百到几千元/年），现在有了 **Let's Encrypt**：
- 完全免费
- 自动化颁发（几分钟搞定）
- 有效期 90 天，但可以自动续期

**这就是为什么现在几乎所有网站都用 HTTPS 了。**

### 自签名证书：自己给自己发的"身份证"

你也可以自己生成证书（不找 CA），叫"自签名证书"。

**问题**：浏览器不信任，会报警告：

```
⚠️ 您的连接不是私密连接
攻击者可能会试图窃取您的信息...
```

**什么时候用自签名证书？**
- 内网测试（反正只有自己用）
- 学习练习

**生产环境千万别用**，用户看到警告会吓跑的。

### ⚠️ 浏览器提示"证书不安全"的常见原因

| 原因 | 说明 |
|------|------|
| 证书过期 | 超过有效期了 |
| 域名不匹配 | 证书是给 `a.com` 的，你访问的是 `b.com` |
| 自签名证书 | 不是 CA 签发的 |
| CA 不被信任 | 证书是某个小 CA 签的，浏览器不认识 |

### 💻 动手验证：查看网站证书

**在浏览器里**（以 Chrome 为例）：
1. 访问任意 HTTPS 网站（比如 `https://www.baidu.com`）
2. 点地址栏左边的锁图标
3. 点"连接是安全的" → "证书有效"
4. 查看证书详情

你会看到：
- 颁发给谁
- 谁颁发的
- 有效期
- 公钥信息

**在终端里**（查看证书）：

```bash
# Mac / Linux
openssl s_client -connect www.baidu.com:443 -showcerts < /dev/null

# 只看证书摘要
openssl s_client -connect www.baidu.com:443 < /dev/null 2>/dev/null | openssl x509 -noout -text
```

输出会很长，重点看：
- `Subject:` 颁发给谁
- `Issuer:` 谁颁发的
- `Validity:` 有效期

## 5. SSH 密钥认证：比密码更安全的登录方式

### 🔑 公钥放服务器，私钥留本地

回顾第 3 章，SSH 登录服务器有两种方式：
1. **密码认证**：输入用户名密码
2. **密钥认证**：用密钥对（公钥 + 私钥）

**密钥认证更安全，为什么？**

### 登录过程（简化版）

```
1. 你：我要登录，我的用户名是 alice
2. 服务器：我这里有 alice 的公钥，我用它加密一个随机数发给你
3. 你：我用私钥解密，把结果发回去
4. 服务器：对上了，证明你是 alice，放行
```

**关键点**：
- 私钥文件不会在网络上传输
- 即使有人截获数据包，也没有私钥，解不开

### 为什么比密码安全？

| 密码认证 | 密钥认证 |
|---------|---------|
| 密码可以被猜（暴力破解） | 私钥文件有 2048 位，猜不出来 |
| 密码可能被截获（中间人攻击） | 私钥不在网络上传输 |
| 密码可能被钓鱼（假网站） | 密钥对应服务器，钓鱼网站没用 |

### 💻 动手验证：生成密钥对

**生成密钥对**：

```bash
# Mac / Linux
# 推荐：Ed25519 算法（更安全、密钥更短）
ssh-keygen -t ed25519 -C "your_email@example.com"

# 备选：如果系统不支持 Ed25519，用 RSA 4096
# ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

# 解释：
# -t ed25519：使用 Ed25519 算法（现代推荐）
# -t rsa -b 4096：使用 RSA 算法，密钥长度 4096 位
# -C：注释（通常写邮箱，方便识别）
```

**交互过程**：

```
Enter file in which to save the key (/home/user/.ssh/id_ed25519):
# 直接回车，用默认路径（RSA 则为 id_rsa）

Enter passphrase (empty for no passphrase):
# 可以设置密码保护私钥，也可以直接回车（不设密码）

Enter same passphrase again:
# 再输一次密码（或直接回车）
```

**生成的文件**：

```
~/.ssh/id_ed25519       # 私钥（千万别泄露）
~/.ssh/id_ed25519.pub   # 公钥（可以公开）
# 如果用 RSA，则为 id_rsa 和 id_rsa.pub
```

### 查看公钥内容

```bash
# Mac / Linux
cat ~/.ssh/id_ed25519.pub  # 或 id_rsa.pub（如果用 RSA）
```

输出类似：

```
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... your_email@example.com
# 如果用 RSA，则以 ssh-rsa 开头
```

这一长串就是公钥，可以放心复制。

### 部署公钥到服务器

**方法 1：用 `ssh-copy-id`（推荐）**

```bash
# Mac / Linux
ssh-copy-id user@server_ip

# 会提示输入密码（最后一次用密码）
# 之后就可以免密登录了
```

**方法 2：手动复制**

```bash
# 1. 复制公钥内容
cat ~/.ssh/id_ed25519.pub  # 或 id_rsa.pub（如果用 RSA）

# 2. 登录服务器（用密码）
ssh user@server_ip

# 3. 在服务器上，把公钥追加到 authorized_keys
mkdir -p ~/.ssh
echo "刚才复制的公钥内容" >> ~/.ssh/authorized_keys
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

### ⚠️ 私钥文件权限必须是 600

SSH 对私钥文件的权限要求很严格：

```bash
# 查看权限
ls -l ~/.ssh/id_ed25519  # 或 id_rsa（如果用 RSA）

# 应该是：
-rw-------  1 user user 419 Jan 01 12:00 id_ed25519
# 只有所有者能读写（600）
```

**如果权限不对**（比如 644），SSH 会拒绝使用：

```
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@         WARNING: UNPROTECTED PRIVATE KEY FILE!          @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
Permissions 0644 for '/home/user/.ssh/id_ed25519' are too open.
```

**修复方法**：

```bash
# Mac / Linux
chmod 600 ~/.ssh/id_ed25519  # 或 id_rsa（如果用 RSA）
```

### 💻 验证密钥登录

```bash
# Mac / Linux
ssh user@server_ip

# 如果配置正确，直接登录，不需要输密码
```

**调试模式**（如果登录失败）：

```bash
# Mac / Linux
ssh -v user@server_ip

# -v：显示详细日志，看哪里出错了
```

## 6. VPN 的加密原理

### 🔑 VPN：加密隧道

**一句话概括**：VPN 在你和服务器之间建立一条加密隧道，所有数据在隧道里传输。

**生活类比**：
- **不用 VPN**：你在大街上走，所有人都能看到你去了哪里
- **用 VPN**：你钻进地铁隧道，外面的人只知道你进了隧道，不知道你在隧道里干什么

### VPN 怎么工作？

```
你的电脑 → [加密] → VPN 服务器 → [解密] → 目标网站
         ←         ←              ←
```

**关键点**：
1. 数据在进隧道前加密
2. 中间的运营商、路由器只能看到加密后的数据
3. 数据到达 VPN 服务器后解密，再转发给目标网站

> ⚠️ **"隧道"在网络中有多种含义：**
> 
> | 场景 | 含义 | 例子 |
> |------|------|------|
> | VPN 隧道 | 加密的虚拟通道，保护所有流量 | WireGuard、OpenVPN |
> | SSH 隧道 | 通过 SSH 连接转发特定端口的流量 | `ssh -L 8080:localhost:80 server` |
> | 代理隧道 | 通过代理服务器中转流量 | Shadowsocks、V2Ray |
> | 隧道协议 | 把一种协议封装在另一种协议里传输 | GRE、IPSec |
> 
> 共同点：都是"在已有网络上建立一条虚拟的专用通道"。区别在于加密方式、工作层次和用途不同。


**运营商看到的**：
- 你在和某个 IP（VPN 服务器）通信
- 数据是加密的，看不懂内容
- 不知道你在访问什么网站

### 什么时候需要 VPN？

| 场景 | 为什么需要 |
|------|-----------|
| 公共 Wi-Fi | 防止被截获数据 |
| 访问公司内网 | 远程办公，安全连接 |
| 绕过地域限制 | 访问某些被限制的服务 |
| 隐私保护 | 不想让运营商知道你在访问什么 |

### 常见 VPN 协议对比

| 协议 | 特点 | 适用场景 |
|------|------|---------|
| **WireGuard** | 新、快、配置简单 | 个人用户，推荐 |
| **OpenVPN** | 老牌、稳定、配置复杂 | 企业、需要兼容性 |
| **IPSec** | 企业常用，配置最复杂 | 大型企业内网 |

⚠️ **不推荐**：PPTP（老旧，不安全）

### 你现在用的"VPN 订阅链接"是什么？

> ⚠️ **术语澄清：** 很多人把 Clash、V2Ray 等翻墙工具叫做"VPN"，但它们技术上是**加密代理工具**，不是真正的 VPN。区别在于：VPN 在网络层加密所有流量，代理工具在应用层按规则转发流量。详细对比见 [第 8 章 · 8.3 节](09_remote_access.md)。

这些翻墙工具（如 Clash、V2Ray、Shadowrocket）导入一个"订阅链接"就能用。

**订阅链接的本质是什么？**

一个包含以下信息的配置文件集合（通常包含多个"节点"）：
- 服务器地址（IP 或域名）
- 端口号
- 加密方式（比如 AES-256-GCM）
- 密钥或密码
- 协议类型（Shadowsocks、VMess、Trojan 等）

客户端读取配置后，自动连接服务器，建立加密通道。

**订阅链接通常是 Base64 编码的**，解码后就是明文配置。

> 💡 **想深入了解翻墙工具的工作原理？** 包括机场、节点、协议区别、TUN 模式等概念，请看 [第 8 章 · 8.6 节 翻墙工具全景图](09_remote_access.md)。

### 💻 动手验证：查看 VPN 连接状态

**Mac**：

```bash
# 查看网络接口（VPN 连接后会多一个 utun 接口）
ifconfig | grep utun

# 查看路由表（VPN 会修改路由）
netstat -nr
```

**Linux**：

```bash
# 查看网络接口
ip addr show

# 查看路由表
ip route show
```

**连接 VPN 前后对比**：
- 连接前：默认网关是你的路由器
- 连接后：默认网关变成 VPN 服务器

## 7. 本章小结

### 核心概念回顾

| 概念 | 一句话总结 |
|------|-----------|
| **HTTP vs HTTPS** | HTTP 是明信片，HTTPS 是密封信 |
| **对称加密** | 一把钥匙，快但不安全传输 |
| **非对称加密** | 两把钥匙（公钥+私钥），解决钥匙传输问题 |
| **TLS/SSL** | HTTPS 背后的加密机制 |
| **证书** | 网站的"身份证"，防止冒充 |
| **CA** | 发证书的"公安局" |
| **SSH 密钥** | 比密码更安全的登录方式 |
| **VPN** | 加密隧道，隐藏你的网络流量 |

### 你现在能做什么？

✅ 理解为什么 HTTPS 比 HTTP 安全
✅ 知道加密的基本原理（对称 vs 非对称）
✅ 会生成 SSH 密钥对并配置免密登录
✅ 能看懂浏览器里的证书信息
✅ 明白 VPN 的加密原理

### ⚠️ 易错点总结

1. **私钥文件权限必须是 600**，否则 SSH 拒绝使用
2. **自签名证书不能用于生产环境**，浏览器会报警告
3. **VPN 不是匿名工具**，VPN 服务商能看到你的流量
4. **HTTPS 不是万能的**，只保护传输过程，不保护服务器本身

### 3. 常见问题

**Q：HTTPS 会不会被破解？**
A：理论上可以，但需要几百年的计算时间。实际上不可能。

**Q：VPN 能保证完全匿名吗？**
A：不能。VPN 服务商能看到你访问的网站。

```
VPN 加密范围：

你的电脑 ──加密隧道──→ VPN服务器 ──解密──→ 目标网站
                        ↑
                   这里数据是明文！
```

VPN 只加密"你到 VPN 服务器"这一段，VPN 服务器到目标网站是明文的。

**VPN 服务商能看到什么？**

| 内容 | 能看到吗 | 原因 |
|------|---------|------|
| 你访问的网站（IP/域名） | 能 | DNS 查询和目标 IP |
| HTTP 网站的内容 | 能 | HTTP 是明文 |
| HTTPS 网站的内容 | 不能 | HTTPS 是端到端加密，只看到域名 |

**VPN vs Tor（洋葱路由）**

| | VPN | Tor |
|---|---|---|
| **原理** | 一条加密隧道 | 三层加密，三个节点 |
| **谁知道你访问了什么** | VPN 服务商知道 | 没有任何节点知道完整路径 |
| **速度** | 快 | 慢（绕三圈） |
| **适用场景** | 日常使用 | 需要高度匿名 |

```
Tor（洋葱路由）的工作方式：

你 → 节点1 → 节点2 → 节点3 → 目标网站
     ↑        ↑        ↑
   只知道    只知道    只知道
   你和节点2  节点1和3  节点2和目标

- 节点1：知道你是谁，但不知道你访问什么
- 节点2：什么都不知道（只转发加密数据）
- 节点3：知道访问什么，但不知道你是谁

没有任何一个节点知道完整路径！
```

**结论**：
- VPN：隐藏你的 IP，但 VPN 服务商知道你访问了什么
- Tor：真正匿名，但速度慢，适合需要高度隐私的场景

**Q：为什么有的网站是 HTTP 和 HTTPS 都能访问？**
A：服务器同时开了 80（HTTP）和 443（HTTPS）端口。建议强制跳转到 HTTPS。

**Q：SSH 密钥丢了怎么办？**
A：重新生成一对，部署新公钥到服务器。旧密钥就作废了。

### 4. 进阶话题

**零信任网络（Zero Trust）**：
- 传统安全：内网可信，外网不可信
- 零信任：任何网络都不可信，每次访问都要验证

**量子计算对加密的威胁**：
- 量子计算机可以破解现有的 RSA、ECC
- 正在研究"抗量子加密算法"

**区块链中的加密**：
- 比特币用的是 ECDSA（椭圆曲线数字签名）
- 私钥丢了，币就永远找不回来了

### 5. 命令速查表

```bash
# ===== 证书相关 =====
# 查看网站证书
openssl s_client -connect example.com:443 -showcerts < /dev/null

# 查看证书文件内容
openssl x509 -in cert.pem -text -noout

# 生成自签名证书（测试用）
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes

# ===== SSH 密钥相关 =====
# 生成密钥对（推荐 ed25519）
ssh-keygen -t ed25519 -C "your_email@example.com"

# 查看公钥
cat ~/.ssh/id_ed25519.pub  # 或 id_rsa.pub（如果用 RSA）

# 部署公钥到服务器
ssh-copy-id user@server_ip

# 修改私钥权限
chmod 600 ~/.ssh/id_ed25519  # 或 id_rsa（如果用 RSA）

# 调试 SSH 连接
ssh -v user@server_ip

# ===== VPN 相关 =====
# 查看网络接口（Mac）
ifconfig | grep utun

# 查看网络接口（Linux）
ip addr show

# 查看路由表（Mac）
netstat -nr

# 查看路由表（Linux）
ip route show

# ===== 加密测试 =====
# 用 openssl 加密文件（对称加密）
openssl enc -aes-256-cbc -salt -in file.txt -out file.txt.enc

# 解密文件
openssl enc -aes-256-cbc -d -in file.txt.enc -out file.txt

# 生成随机密码
openssl rand -base64 32
```
