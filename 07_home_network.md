## 7.2 家庭网络拓扑

### 标准家庭网络长这样

```
互联网
  │
  ↓
┌──────────┐  光纤   ┌──────────┐  网线    ┌──────────┐
│  运营商   │ ──────→ │   光猫    │ ───────→ │  路由器   │
│          │         │          │          │          │
└──────────┘         └──────────┘          └──────────┘
                                             │  │  │
                                    ┌────────┘  │  └────────┐
                                    ↓           ↓           ↓
                                  📱手机     💻电脑      🖥️NAS
                                 (Wi-Fi)    (Wi-Fi/网线)  (网线)
```

### 光猫的两种模式

| 模式 | 光猫做什么 | 路由器做什么 | 推荐？ |
|---|---|---|---|
| **路由模式**（默认） | 拨号 + NAT + DHCP + Wi-Fi | 再做一次 NAT + DHCP | ❌ 双重 NAT |
| **桥接模式**（推荐） | 只做光电转换 | 拨号 + NAT + DHCP + Wi-Fi | ✅ 推荐 |

### ⚠️ 双重 NAT 的问题

如果光猫是路由模式，你的网络就有**两层 NAT**：

```
互联网 → 光猫(NAT第1层) → 路由器(NAT第2层) → 你的设备
```

问题：
- 端口转发配置复杂（两层都要配）
- 某些游戏/应用可能连接异常
- 性能略有损失

🔑 **建议：打电话给运营商，让他们把光猫改成桥接模式。** 一般说"我要改桥接"客服就懂了。有些需要超级管理员密码，可以问客服要。

> 💡 **如果运营商不给改桥接也能用：** 先继续用路由模式，不影响日常上网；远程访问场景建议直接用内网穿透方案（7.9 节）。

## 7.3 路由器初始设置

### 第 1 步：进入路由器后台

```
浏览器输入路由器管理地址（通常在路由器底部标签上）：
- 常见地址：192.168.1.1 或 192.168.0.1 或 10.0.0.1
- 常见用户名/密码：admin/admin 或 admin/password
```

⚠️ **第一件事：改掉默认密码！** 否则同一 Wi-Fi 下的任何人都能进你的路由器后台。

### 第 2 步：配置 WAN 口（上网方式）

WAN 口就是路由器连接外网的接口。根据你的情况选择：

| 上网方式 | 什么时候用 | 需要填什么 |
|---|---|---|
| **PPPoE 拨号** | 光猫桥接模式（最常见） | 运营商给的账号和密码 |
| **DHCP 自动获取** | 光猫是路由模式 | 什么都不用填 |
| **静态 IP** | 企业专线 | IP、子网掩码、网关、DNS |

🔑 **家庭用户 90% 的情况：光猫桥接 + 路由器 PPPoE 拨号。**

### 第 3 步：配置 LAN 口

LAN 口是路由器面向内网的设置：

```
路由器 IP：192.168.1.1（这就是你的网关）
子网掩码：255.255.255.0
```

一般默认就行，不需要改。

## 7.4 Wi-Fi 配置

### 基本设置

| 设置项 | 建议 |
|---|---|
| **SSID（名称）** | 起个你能认出来的名字，别用默认的 |
| **密码** | 至少 12 位，混合大小写数字 |
| **加密方式** | WPA3（最新最安全）> WPA2 > WPA > WEP（别用）|
| **隐藏 SSID** | 不建议，增加连接麻烦但安全性提升有限 |

### 2.4G vs 5G

| 特性 | 2.4 GHz | 5 GHz |
|---|---|---|
| **速度** | 较慢（最高约 300Mbps） | 较快（最高约 1733Mbps+） |
| **穿墙** | 好 | 差 |
| **干扰** | 多（微波炉、蓝牙都用 2.4G） | 少 |
| **适合** | 远距离、IoT 设备 | 近距离、看视频、打游戏 |

🔑 **建议：2.4G 和 5G 分开命名**（比如 `MyHome` 和 `MyHome_5G`），这样你能手动选择连哪个。

有些路由器支持"双频合一"（Band Steering），自动帮你选——但有时候选得不好，建议分开。

### 信道选择

```
2.4G 推荐信道：1、6、11（互不干扰的三个）
5G 推荐信道：一般自动就行
```

💻 **查看周围 Wi-Fi 信道占用：**
```bash
# Mac
/System/Library/PrivateFrameworks/Apple80211.framework/Versions/Current/Resources/airport -s

# Linux
sudo iwlist wlan0 scan | grep -E "Channel|ESSID"
```

如果你家周围很多人用信道 6，你就换到 1 或 11。

## 7.5 DHCP 配置

### 地址池设置

```
路由器后台 → DHCP 设置：
  起始地址：192.168.1.100
  结束地址：192.168.1.200
  租约时间：24 小时（默认就行）
```

🔑 **留出一段 IP 给静态设备用：** 比如 192.168.1.2~99 留给手动配置，100~200 给 DHCP 自动分配。

### 静态 IP 绑定（DHCP Reservation）

让某些设备永远拿到同一个 IP，比如 NAS、打印机、服务器。

#### 方法1：DHCP静态绑定（推荐）

**原理**：在路由器上设置，让路由器记住这台设备的MAC地址，每次都给它分配同一个IP。

**步骤：**

```
第1步：找到电脑的MAC地址

Windows：
  打开命令提示符 → ipconfig /all
  找到"物理地址"，比如：A1-B2-C3-D4-E5-F6

Mac：
  打开终端 → ifconfig
  找到 en0 的 ether，比如：a1:b2:c3:d4:e5:f6

Linux：
  终端 → ip addr
  找到 link/ether 后面的地址
```

```
第2步：在路由器上设置绑定

路由器后台 → DHCP设置 → 静态绑定/地址保留/IP与MAC绑定：

  添加规则：
    设备名称：我的电脑
    MAC地址：A1:B2:C3:D4:E5:F6
    IP地址：192.168.1.50

  保存
```

```
第3步：让电脑重新获取IP

Windows：ipconfig /release 然后 ipconfig /renew
Mac/Linux：断开重连网络，或重启

之后这台电脑就会一直获取 192.168.1.50 这个IP
```

#### 方法2：手动设置静态IP（不推荐）

**原理**：在电脑上直接指定IP，不向路由器请求。

**Windows设置：**
```
控制面板 → 网络和共享中心 → 更改适配器设置
  → 右键网卡 → 属性
  → Internet协议版本4(TCP/IPv4) → 属性
  → 选择"使用下面的IP地址"：

    IP地址：192.168.1.50
    子网掩码：255.255.255.0
    默认网关：192.168.1.1（路由器IP）

    DNS服务器：192.168.1.1 或 114.114.114.114
```

**Mac设置：**
```
系统偏好设置 → 网络 → 选择网卡 → 高级 → TCP/IP

  配置IPv4：手动
  IPv4地址：192.168.1.50
  子网掩码：255.255.255.0
  路由器：192.168.1.1
```

#### 两种方法对比

| | DHCP静态绑定 | 手动设置静态IP |
|---|---|---|
| **在哪设置** | 路由器 | 电脑 |
| **优点** | 集中管理，换网络不影响 | 不依赖路由器 |
| **缺点** | 需要路由器支持 | 换网络要重新配，可能IP冲突 |
| **推荐** | ✅ 推荐 | 不推荐 |

🔑 **推荐用方法1（DHCP静态绑定）**，因为集中在路由器管理，电脑带到别的网络不受影响。

🔑 **为什么要绑定？** 因为端口转发、防火墙规则都是基于 IP 的。如果 NAS 的 IP 每次都变，规则就失效了。

## 7.6 从外网访问内网 —— 先判断，再选方案

### 问题背景：NAT 的单向性

在第 4 章我们学过 NAT：你家路由器把内网 IP 转换成公网 IP 才能上网。

NAT 的特点是**单向的**——内网可以主动访问外网，但外网**不能主动访问内网**。

```
内网设备 → NAT → 互联网    ✅ 可以（内网主动发起）
互联网 → NAT → 内网设备    ❌ 不行（NAT 不知道该转给谁）
```

所以你想从外面访问家里的 NAS，需要一些"穿透"手段。

### 第一步：判断你有没有公网 IP

#### 先搞懂：IPv4 vs IPv6

**现实情况：** 现在很多家庭宽带是 **IPv4 没有公网 IP，但有公网 IPv6**。

```
常见的家庭宽带 IP 情况：

┌─────────────────────────────────────────────────────────────────┐
│  IPv4                           IPv6                            │
│  ─────                          ─────                           │
│  地址不够用，运营商要 NAT        地址足够，每个设备都能有公网地址  │
│  大部分家庭没有公网 IPv4         大部分家庭有公网 IPv6            │
│  10.x.x.x 或 100.64.x.x = NAT   240e:xxxx:xxxx = 公网 IPv6      │
└─────────────────────────────────────────────────────────────────┘
```

你可以这样查看自己的情况：

```bash
# 查看 IPv4
curl -4 ifconfig.me
# 如果返回 10.x.x.x 或 100.64.x.x → IPv4 没有

# 查看 IPv6
curl -6 ifconfig.me
# 如果返回 240e:xxxx 或 2408:xxxx 等 → IPv6 有（240e 是电信，2408 是联通）
```

#### IPv4 判断

```bash
# 第 1 步：查看路由器 WAN 口 IPv4（路由器后台 → 网络状态 / WAN 设置）
# 记下这个 IP，比如：123.45.67.89 或 10.12.34.56

# 第 2 步：查看你对外显示的公网 IPv4
curl -4 ifconfig.me
# 或访问浏览器 https://ip.sb

# 第 3 步：对比两个 IP
```

**IPv4 判断结果：**

| 对比结果 | 结论 | 你的方案 |
|---------|------|---------|
| 两个 IP **一样**，且不是 10.x / 100.64.x | ✅ 有公网 IPv4 | **7.7 端口转发** |
| 两个 IP **一样**，但过几天变了 | ✅ 有公网 IPv4（动态） | **7.7 端口转发 + 7.8 DDNS** |
| 两个 IP **不一样** | ❌ 没有公网 IPv4 | 看 IPv6 或用内网穿透 |
| WAN 口是 10.x.x.x 或 100.64~127.x.x | ❌ 运营商级 NAT（CGNAT） | 看 IPv6 或用内网穿透 |

#### IPv6 判断

```bash
# 查看本机 IPv6
ip addr | grep inet6    # Linux
ifconfig | grep inet6   # Mac

# 查看对外 IPv6
curl -6 ifconfig.me
```

**IPv6 判断结果：**

| 情况 | 结论 | 说明 |
|-----|------|------|
| 有 `240e:` / `2408:` / `2409:` 开头的地址 | ✅ 有公网 IPv6 | 可以直接用 IPv6 访问 |
| 只有 `fe80:` 开头的地址 | ❌ 只有本地链路地址 | 没有 IPv6 公网 |
| 只有 `fd` / `fc` 开头的地址 | ❌ 内网 IPv6 | 没有 IPv6 公网 |

> ⚠️ **注意：** 即使有公网 IPv6，路由器的 IPv6 防火墙默认可能会**阻止入站连接**。想从外网访问，需要在路由器上配置 IPv6 防火墙规则，允许特定端口入站。

#### 四种情况，四条路

```
你想从外网访问家里设备
        │
        ├── 有公网 IPv4（固定）
        │       └──→ 7.7 端口转发（最简单）
        │
        ├── 有公网 IPv4（动态，会变）
        │       └──→ 7.7 端口转发 + 7.8 DDNS
        │
        ├── 没有公网 IPv4，但有公网 IPv6
        │       │
        │       ├── 访问者也支持 IPv6？
        │       │       └──→ 用 IPv6 地址访问（需配置路由器 IPv6 防火墙）
        │       │
        │       └── 访问者不支持 IPv6？
        │               └──→ 7.9 内网穿透
        │
        └── IPv4 和 IPv6 都没有公网
                │
                ├── 自己几台设备互联？
                │       └──→ 7.9.3 Tailscale（最简单）
                │
                ├── 有 VPS，想要灵活控制？
                │       └──→ 7.9.2 frp（最灵活）
                │
                ├── 暴露 Web 服务，有域名？
                │       └──→ 7.9.4 Cloudflare Tunnel
                │
                └── 临时测试？
                        └──→ 7.9.5 ngrok
```

### 如果没有公网 IPv4，能怎么办？

1. **检查有没有公网 IPv6**（上面说过怎么判断）
   - 如果有，且访问者也支持 IPv6，可以直接用 IPv6 地址访问
   - 需要在路由器上配置 IPv6 防火墙，允许入站连接

2. **打电话给运营商申请公网 IPv4**：说"我需要一个公网 IP"或"我要改桥接模式"
   - 有些地区免费给，有些要加钱（企业宽带），有些根本不给
   - 注意：申请的是 IPv4，IPv6 通常已经有了

3. **如果运营商不给**：用内网穿透（7.9 节），不需要公网 IP 也能从外网访问

## 7.7 方案一：端口转发 —— 有公网 IP 时用

### 一句话定义

🔑 **端口转发 = 路由器收到外网请求后，把它"转交"给内网的某台设备。**

### 原理图解

```
外网用户                    你家路由器（公网IP）                内网
                              ┌──────────────┐
  访问 123.45.67.89:5000 ───→ │  NAT 表      │
                              │              │
  ←───── 数据返回 ─────────── │  外部端口5000 │ ───转发───→ NAS 192.168.1.50:5000
                              │     ↓        │
                              │  转发到      │
                              │  内部IP:端口  │
                              └──────────────┘

数据流向：
外网用户 → 路由器公网IP:5000 → NAT查表转发 → 内网设备192.168.1.50:5000
```

### 配置步骤

#### 第 1 步：确保内网设备 IP 固定

端口转发规则是基于 IP 的，如果内网设备 IP 变了，规则就失效。所以先在路由器上做 DHCP 静态绑定（见 7.5 节），让 NAS 的 IP 固定为 192.168.1.50。

#### 第 2 步：在路由器上配置端口转发

```
路由器后台 → 找到"端口转发" / "虚拟服务器" / "NAT设置"：

  添加规则：
    服务名称：NAS-Web
    外部端口：5000（外网访问的端口）
    内部 IP：192.168.1.50（NAS 的 IP）
    内部端口：5000（NAS 服务的端口）
    协议：TCP
```

**不同品牌路由器界面名称不同：**
- TP-Link：虚拟服务器
- 华硕：端口转发
- 小米：端口转发
- OpenWrt：Firewall → Port Forwards

#### 第 3 步：测试

```bash
# 在内网测试（确保服务本身正常）
curl http://192.168.1.50:5000

# 在外网测试（用手机 4G 网络或朋友家网络）
curl http://你的公网IP:5000
# 或浏览器访问 http://你的公网IP:5000
```

### 完整示例：从外网 SSH 连家里电脑

```
第 1 步：DHCP 绑定
  路由器后台 → DHCP → 静态绑定
  电脑 MAC → 192.168.1.100

第 2 步：端口转发
  路由器后台 → 端口转发
  外部端口：22222（不要用 22，不安全）
  内部 IP：192.168.1.100
  内部端口：22
  协议：TCP

第 3 步：确保电脑开启了 SSH
  Windows：设置 → 应用 → 可选功能 → OpenSSH 服务器
  Mac：系统偏好设置 → 共享 → 远程登录
  Linux：sudo apt install openssh-server

第 4 步：从外网连接
  ssh -p 22222 用户名@你的公网IP
```

### ⚠️ 安全提醒

- **不要转发 22 端口（SSH）到公网**，会被全球扫描器暴力破解
- 如果必须转发 SSH，改成高端口（如 22222）并用密钥登录
- 能用内网穿透就不要用端口转发，更安全

## 7.8 方案二：DDNS —— 解决公网 IP 变化问题

### 问题：公网 IP 会变

大多数家庭宽带的公网 IP 是**动态的**，每次重启光猫或过一段时间就会变。

```
今天你告诉朋友：访问 123.45.67.89:5000 就能看到我的 NAS
明天 IP 变了：123.45.67.89 已经不是你的了，朋友访问不了
```

### 解决方案：DDNS（动态域名）

DDNS 服务会给你一个固定域名（如 `myhome.ddns.net`），**自动更新**指向你当前的公网 IP。

```
┌─────────────────────────────────────────────────────────────┐
│  你的路由器                 DDNS 服务器                      │
│     │                          │                            │
│     │ 1. 检测到 IP 变化         │                            │
│     │ 2. 上报新 IP ──────────→ │                            │
│     │                          │ 3. 更新域名解析              │
│     │                          │    myhome.ddns.net → 新IP   │
│     │ ←────────────────────── │                            │
│     │                    4. 确认更新                         │
└─────────────────────────────────────────────────────────────┘

朋友访问 myhome.ddns.net:5000 → 自动解析到你当前 IP → 端口转发到 NAS
```

### DDNS 和端口转发的关系

| | 端口转发 | DDNS |
|--|---------|------|
| **解决什么问题** | 让外网请求能到达内网设备 | 解决 IP 变化问题 |
| **需要公网 IP** | ✅ 必须 | ✅ 必须（DDNS 不解决无公网 IP 的问题） |
| **一起用？** | ✅ 通常配合使用 | ✅ 配合端口转发 |

🔑 **结论：有动态公网 IP → 端口转发（7.7）+ DDNS（7.8）一起用。**

### 常用 DDNS 服务

| 服务 | 免费？ | 说明 |
|---|---|---|
| 路由器自带 | ✅ | 很多路由器内置花生壳、NO-IP 等 |
| Cloudflare API | ✅ | 需要自己的域名，用脚本更新 |
| DuckDNS | ✅ | 免费简单 |
| 花生壳 | 部分免费 | 国内常用 |

### 配置方法（以路由器自带为例）

```
路由器后台 → DDNS 设置：
  服务商：NO-IP / 花生壳 / DynDNS
  域名：myhome.ddns.net
  用户名：xxx
  密码：xxx
  → 启用
```

配置后，用 `myhome.ddns.net:5000` 就能从外网访问你的 NAS，不用记 IP。

## 7.9 方案三：内网穿透 —— 没有公网 IP 时用

### 7.9.1 原理

既然外网不能主动连内网，那就让**内网主动连外网**，建立一个隧道：

```
内网设备主动连 ──→ 中继服务器（有公网IP）←── 外网用户来连
                        ↑
                 这是双方都能访问的"中间人"
```

🔑 **内网穿透的本质：找一个双方都能访问的"中间人"，内网主动连中间人，外网也连中间人，中间人负责转发数据。**

```
外网用户                    中继服务器（公网）              你家内网
                           ┌──────────────┐
  访问服务器 ─────────────→ │              │
                           │   中间人      │ ←──── 主动连 ──── 内网客户端
  ←──── 数据返回 ────────── │              │ ────── 转发 ────→ 内网设备
                           └──────────────┘

关键点：内网客户端"主动"连出去（NAT 允许），连接保持不断，
       外网用户访问服务器时，服务器通过这个连接把数据转给内网。
```

### 7.9.2 frp —— 自建内网穿透（最灵活）

**适合：** 有自己的 VPS，想要灵活控制，支持任意 TCP/UDP 协议

#### 原理

frp 需要一台有公网 IP 的服务器（VPS）做中转：

```
外网用户                    你的 VPS（公网）              你家内网
                           ┌──────────────┐
  访问 VPS:6000  ────────→ │   frps       │
                           │  (服务端)     │ ←──── 长连接 ──── frpc (客户端)
  ←──── 数据返回 ────────── │              │ ────── 转发 ────→ 内网设备:22
                           └──────────────┘
```

#### 服务端配置（VPS 上）

```bash
# 第 1 步：下载 frp
wget https://github.com/fatedier/frp/releases/download/v0.58.0/frp_0.58.0_linux_amd64.tar.gz
tar -xzf frp_0.58.0_linux_amd64.tar.gz
cd frp_0.58.0_linux_amd64
```

```toml
# frps.toml（服务端配置）
bindPort = 7000           # frps 监听端口
auth.token = "REPLACE_WITH_STRONG_RANDOM_TOKEN"  # 认证 token，客户端要一致

# Web 管理面板（可选）
webServer.addr = "127.0.0.1"
webServer.port = 7500
webServer.user = "admin"
webServer.password = "REPLACE_WITH_STRONG_PASSWORD"
```

⚠️ **安全提醒：** `auth.token` 和 `webServer.password` 必须替换成强随机字符串，不要用弱口令。

```bash
# 第 2 步：启动 frps
./frps -c frps.toml

# 第 3 步：设为系统服务（开机自启）
sudo mkdir -p /opt/frp
sudo cp frps frps.toml /opt/frp/
sudo vim /etc/systemd/system/frps.service
```

```ini
[Unit]
Description=frps service
After=network.target

[Service]
Type=simple
ExecStart=/opt/frp/frps -c /opt/frp/frps.toml
Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable frps
sudo systemctl start frps

# 开防火墙
sudo ufw allow 7000/tcp    # frp 通信端口
sudo ufw allow 6000/tcp    # SSH 穿透端口
```

#### 客户端配置（家里的电脑/NAS 上）

```toml
# frpc.toml（客户端配置）
serverAddr = "你的VPS的IP"
serverPort = 7000
auth.token = "REPLACE_WITH_STRONG_RANDOM_TOKEN"  # 和服务端一致

# 穿透 SSH
[[proxies]]
name = "home-ssh"
type = "tcp"
localIP = "127.0.0.1"
localPort = 22                 # 内网设备的 SSH 端口
remotePort = 6000              # VPS 上暴露的端口
```

```bash
# 启动 frpc
./frpc -c frpc.toml

# 测试：从外网 SSH 连接
ssh -p 6000 用户名@你的VPS的IP
# 实际连接的是你家里的电脑 ✅
```

### 7.9.3 Tailscale —— 零配置组网（✅ 最简单）

**适合：** 自己的几台设备互相访问，不想管服务器

#### 一句话概括

🔑 **Tailscale 是一种 Mesh VPN（网状 VPN），让你的所有设备自动组成一个私有网络，不需要公网 IP，不需要自己搭服务器。**

> ⚠️ **注意：** Tailscale 创建的是 **Overlay Network（覆盖网络）/ Mesh VPN**，不是第 4 章说的 **VLAN**。VLAN 是交换机上的逻辑分组（同一地点），Tailscale 是把不同地点的设备连起来。

#### 原理

```
你的 Mac ──→ Tailscale 协调服务器 ←── 你家的 NAS
    │              │                      │
    │    交换连接信息（不传数据）           │
    │                                     │
    └──── WireGuard 直连（P2P）────────────┘
          100.x.x.x ←→ 100.y.y.y
```

- Tailscale 的服务器只负责**协调连接**（交换密钥和地址）
- 实际数据走 **P2P 直连**（WireGuard 加密），不经过 Tailscale 服务器
- 如果 P2P 打不通，才走 Tailscale 的中继服务器

#### 安装和使用

```bash
# Mac
brew install tailscale

# Linux
curl -fsSL https://tailscale.com/install.sh | sh

# 启动并登录
sudo tailscale up
# 会弹出浏览器让你登录（支持 Google/GitHub/Microsoft 账号）

# 查看分配的 IP
tailscale ip -4
# 输出类似：100.64.0.1

# 查看网络中的所有设备
tailscale status
# 输出：
# 100.64.0.1    my-mac        你的Mac
# 100.64.0.2    home-nas      家里的NAS
# 100.64.0.3    work-server   工作服务器
```

#### 使用场景

```bash
# 在外面 SSH 连家里的 NAS（用 Tailscale IP）
ssh admin@100.64.0.2

# 在外面访问家里 NAS 的 Web 界面
# 浏览器打开 http://100.64.0.2:5000
```

#### Tailscale 免费版限制

| 特性 | 免费版 |
|---|---|
| 设备数量 | 100 台 |
| 用户数量 | 3 个 |

🔑 **个人使用完全够了。**

### 7.9.4 Cloudflare Tunnel —— 免费 HTTPS 隧道

**适合：** 暴露 Web 服务给外部访问，有域名托管在 Cloudflare

#### 限制

- ⚠️ **只支持 HTTP/HTTPS**（不能穿透 SSH、TCP 等）
- 需要域名托管在 Cloudflare

#### 配置步骤

```bash
# 第 1 步：安装 cloudflared
# Mac
brew install cloudflared
# Linux
wget https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64
chmod +x cloudflared-linux-amd64
sudo mv cloudflared-linux-amd64 /usr/local/bin/cloudflared

# 第 2 步：登录
cloudflared tunnel login
# 浏览器会打开，选择你的域名授权

# 第 3 步：创建隧道
cloudflared tunnel create my-tunnel

# 第 4 步：配置 DNS
cloudflared tunnel route dns my-tunnel nas.yourdomain.com

# 第 5 步：创建配置文件
vim ~/.cloudflared/config.yml
```

```yaml
# ~/.cloudflared/config.yml
tunnel: <你的隧道ID>
credentials-file: /home/ubuntu/.cloudflared/<隧道ID>.json

ingress:
  - hostname: nas.yourdomain.com
    service: http://localhost:5000
  - hostname: blog.yourdomain.com
    service: http://localhost:8080
  - service: http_status:404    # 默认返回 404
```

```bash
# 第 6 步：启动
cloudflared tunnel run my-tunnel

# 设为系统服务
sudo cloudflared service install
```

现在访问 `https://nas.yourdomain.com` 就能到达你内网的 5000 端口，而且**自动 HTTPS**！

### 7.9.5 ngrok —— 一条命令临时穿透

**适合：** 临时给别人看本地网站、测试 Webhook

```bash
# 安装
brew install ngrok    # Mac
# 或从 https://ngrok.com 下载

# 注册并配置 token（免费）
ngrok config add-authtoken <你的token>

# 一条命令穿透本地 8080 端口
ngrok http 8080

# 输出：
# Forwarding  https://abc123.ngrok-free.app → http://localhost:8080
# 把这个 URL 发给别人就能访问了
```

⚠️ **免费版限制：** 每次启动 URL 都会变，有连接数限制，不适合长期使用。

## 7.10 多路由器组网

家里太大，一个路由器覆盖不了？三种方案：

### 方案 1：AP 模式（✅ 推荐）

```
主路由器 ──网线──→ 第二个路由器（设为 AP 模式）
   │                    │
   │ DHCP/NAT           │ 只做 Wi-Fi 信号发射
   │                    │ 关闭 DHCP
   │                    │ 和主路由同一网段
```

**配置要点：**
- 第二个路由器关闭 DHCP
- 第二个路由器 LAN IP 改成主路由同网段（如 192.168.1.2）
- 网线插 LAN 口（不是 WAN 口）
- Wi-Fi 名称和密码可以和主路由一样（实现漫游）

> 💡 **同名同密码 = 漫游？** 是的。如果两个路由器的 Wi-Fi 名称（SSID）和密码一样，手机会认为是同一个网络，走到信号弱的区域时会自动切换到信号更强的那个。但切换时有 1-2 秒中断（断开再重连），不是真正的"无缝"。Mesh 系统才有真正的无缝漫游。

> 💡 **直接插上也能用？** 是的，如果你只是让第二个路由器下的设备能上网，直接把网线插 WAN 口就行。但这样会形成**双重 NAT**（两个不同网段），导致两边的设备不能互访（比如打印机共享会出问题）。AP 模式让所有设备在同一网段，可以互相访问。

### 方案 2：无线桥接（中继）

```
主路由器 ···Wi-Fi···→ 第二个路由器（无线桥接模式）
```

- 不需要拉网线
- 但速度减半（因为要同时收发）
- 稳定性不如有线

### 方案 3：Mesh 组网（✅ 最省心）

```
Mesh 节点 A ←──自动组网──→ Mesh 节点 B ←──自动组网──→ Mesh 节点 C
```

- 买一套 Mesh 路由器（如 TP-Link Deco、华硕 AiMesh、Linksys Velop）
- 自动组网、自动漫游、统一管理
- 支持有线回程（更稳定）和无线回程

### 三种方案对比

| 特性 | AP 模式 | 无线桥接 | Mesh |
|---|---|---|---|
| 需要拉网线 | ✅ 需要 | ❌ 不需要 | 可选 |
| 速度损失 | 无 | 约 50% | 无/很小 |
| 配置难度 | 中等 | 简单 | 最简单 |
| 稳定性 | ✅ 最好 | 一般 | ✅ 好 |
| 成本 | 低（旧路由器就行） | 低 | 高（要买套装） |
| 无缝漫游 | 需要同名同密码 | 不支持 | ✅ 原生支持 |

🔑 **推荐：** 能拉网线就用 AP 模式（最稳定最便宜），不能拉网线就买 Mesh 套装。

## 7.11 常见问题排查

### 问题 1：连上 Wi-Fi 但上不了网

```
排查步骤：
1. 看 IP 地址 → 169.254.x.x？→ DHCP 失败，重启路由器
2. ping 网关（192.168.1.1）→ 不通？→ 路由器问题
3. ping 8.8.8.8 → 不通？→ WAN 口/拨号问题
4. ping baidu.com → 不通？→ DNS 问题，换 DNS
```

### 问题 2：网速慢

```
排查步骤：
1. 用网线直连路由器测速 → 慢？→ 运营商/WAN 口问题
2. 网线快但 Wi-Fi 慢 → 信道拥挤/距离远/干扰
3. 换 5G 频段试试
4. 检查是否有设备在大量下载/上传
```

### 问题 3：端口转发不生效

```
排查步骤：
1. 确认有公网 IP（curl ifconfig.me vs 路由器 WAN IP）
2. 确认服务在内网能访问（curl 192.168.1.50:5000）
3. 确认端口转发规则正确（IP、端口、协议）
4. 确认服务监听的是 0.0.0.0 不是 127.0.0.1
5. 确认没有双重 NAT（光猫也要配或改桥接）
6. 运营商可能封了 80/443 端口 → 换其他端口
```

## 本章小结

| 操作 | 关键点 |
|---|---|
| 光猫 | 改桥接模式，避免双重 NAT |
| 路由器上网 | 桥接后用 PPPoE 拨号 |
| Wi-Fi | WPA2/WPA3，2.4G 和 5G 分开，选不拥挤的信道 |
| DHCP | 留一段给静态绑定，重要设备绑定固定 IP |
| **远程访问** | **先判断有无公网 IP，再选方案** |
| 端口转发 | 有公网 IP 时用，注意安全 |
| DDNS | 有动态公网 IP 时配合端口转发用 |
| 内网穿透 | 没有公网 IP 时用（Tailscale / frp / CF Tunnel） |
| 多路由器 | 能拉线用 AP 模式，不能拉线用 Mesh |

## 📦 扩展区域

### UPnP（通用即插即用）

路由器上的 UPnP 功能允许内网设备**自动配置端口转发**，不需要手动设置。游戏主机、BT 下载等常用。

⚠️ 安全风险：恶意软件也可以利用 UPnP 打开端口。如果不需要，建议关闭。

### QoS（服务质量）

路由器的 QoS 功能可以给不同设备/应用分配带宽优先级：
- 游戏/视频会议 → 高优先级
- 下载/备份 → 低优先级

适合带宽有限但设备多的家庭。

### IPv6

越来越多的运营商开始分配 IPv6 地址。IPv6 的好处是地址足够多，每个设备都能有公网地址，不需要 NAT。

如果路由器支持，可以在 WAN 设置里开启 IPv6（通常选 DHCPv6 或 SLAAC）。

### 智能家居设备隔离（安全建议）

智能音箱、摄像头、智能插座等 IoT 设备安全性通常较差，厂商可能长期不更新固件。建议把它们隔离到单独的网络：

**方案 1：使用路由器的"访客网络"**
- 大多数路由器支持创建访客 Wi-Fi
- 访客网络与主网络隔离，无法访问内网其他设备
- 把 IoT 设备连到访客网络

**方案 2：使用 VLAN（需要支持的路由器）**
- 创建一个 VLAN 专门给 IoT 设备
- 在防火墙上设置规则：IoT VLAN 只能访问互联网，不能访问内网其他 VLAN

**方案 3：使用独立的小路由器**
- 买一个便宜的小路由器专门给 IoT 设备
- 串在主路由下面（形成双层 NAT，但对 IoT 设备影响不大）

> ⚠️ **为什么需要隔离？**
> - 智能摄像头被入侵后，黑客可以看到你家里的画面
> - 智能音箱被入侵后，可以录音窃听
> - IoT 设备被控制后，可能被用来攻击其他目标（DDoS）
>
> 隔离后，即使某个 IoT 设备被入侵，攻击者也只能在这个隔离网络里，无法访问你的电脑、NAS 等重要设备。
