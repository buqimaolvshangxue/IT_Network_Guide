## 一、现状诊断：八个维度的问题

### 1.1 结构问题

| 问题 | 具体表现 |

| **VPN/代理内容碎片化分布** | 分散在4个章节：第4章（VPN vs 代理原理、TUN、GFW）、第5章（VPN加密原理）、第8章（VPN/代理配置、翻墙工具）、第11章（企业VPN分流），读者永远无法在一处建立完整心智模型 |

### 1.6 本质/深度问题（最核心的缺陷）

**用户原话：** "代理和VPN的基本原理，似乎还是没说到本质上"

**根本原因分析：**

当前repo讲VPN和代理时，集中在"你的IP变没变"这个**表象**。但真正的本质是：

| 维度 | 真VPN（WireGuard/OpenVPN） | 代理工具-系统代理模式（Clash等） | 代理工具-TUN模式（Clash TUN等） |
|------|-----|-----------------|----------------|
| **对路由表做了什么** | 添加路由规则，默认路由指向虚拟网卡 | 什么都没做 | 添加路由规则，类似VPN |
| **对DNS做了什么** | 接管DNS服务器设置 | 什么都没做（DNS泄漏！） | 接管DNS（fake-ip） |
| **流量在哪一层被拦截** | 网络层（第3层），所有IP包 | 应用层（第7层），仅配置的应用 | 网络层捕获 → 应用层处理 |
| **为什么公司域名会挂** | VPN改了DNS服务器，公司内网DNS查不到了 | 通常不会挂 | 代理规则没有bypass公司域名 |

**这张表在当前repo中完全不存在。** 这就是"没说到本质"的根源。

## 二、重构核心原则

### 原则1：物理先于抽象

先让读者知道家里有哪些设备（路由器、光猫、交换机），建立物理拓扑图，再讲 IP/MAC/子网这些抽象概念。

**旧顺序**：第1章 IP/MAC → 第6章 网络设备 ❌（读者在第1章看到"路由器分配IP"时不知道路由器长啥样）

**新顺序**：第1章 网络设备 → 第2章 IP/MAC ✅

### 原则2：一个主题一个权威章节

每个概念只在一处深入讲解，其他地方只做简要引用。

- 正向代理（客户端流量控制）→ 第7章深讲（属于流量控制范畴）
- 反向代理（服务端架构）→ 第10章深讲（属于自建服务/部署范畴）
- SSH隧道 → 只在第9章深讲
- Nginx配置 → 只在第10章深讲
- VPN/代理本质 → 只在第7章深讲
- NAT → 只在第5章深讲

### 原则3：路由表是理解VPN/代理的核心钥匙

路由表必须从"第3章的30行扩展"升级为一个重点小节（约200行），并且在VPN/代理章节中反复使用 `netstat -rn` 的真实输出做对比。

### 原则4：用一个完整案例贯穿全书

**核心案例：家庭用户用笔记本访问Google**

- 第0章：展示完整的直连流程（DHCP → ARP → DNS → 路由表 → TCP → NAT → MAC帧）
- 第7章（新）：展示VPN模式和代理模式下的流程变化
- 第11章：在排障章展示"VPN/代理导致DNS问题"的诊断流程

### 原则5：理论章与实战章分离，但前后呼应

- 第7章讲VPN/代理的**本质和原理**（WHY/WHAT）
- 第9章讲VPN/代理的**搭建和配置**（HOW）
- 读者先理解原理，再动手操作

### 原则6：关键内容不藏在扩展区域

TCP三次握手、ARP工作过程、路由表查找——这些是核心内容，必须在正文中讲，不能放在"📦 扩展区域"。

### 原则7：每章标准教材结构

作为教材，每章必须包含以下标准结构：

```
X.1 这章解决什么问题？（已有）
🚦 新手入口（已有）
🎯 学习目标（新增）——"学完这章你应该能"：列3-5个具体能力点
    例：□ 能说出路由表的三个核心字段  □ 能用 netstat -rn 查看并解读路由表

... 正文内容 ...

🔬 动手验证（每章至少1个实验）
    - 每个实验给出：环境要求、操作步骤、预期输出
    - 例：Ch4："打开终端，运行 netstat -rn，找到default行，那个Gateway就是你的默认网关"
    - 例：Ch7："在Clash系统代理模式下，运行 curl cip.cc 对比有无代理的输出"

📝 本章小结（新增）——3-5条关键takeaway
    - 用"一句话"形式，方便复习
    - 例：Ch7："代理不改路由表，VPN改；TUN模式也改但本质还是代理"
```

注意：当前只有第7章有§7.11本章小结。重构时**所有13章**都需补充学习目标和本章小结。

7.2 【关键】先搞清楚名字——"VPN"不一定是VPN
    ⚠️ 在日常口语中，"开VPN""挂梯子"几乎成了"翻墙"的同义词。

7.4 VPN的本质：修改路由表 + 接管DNS（这里的VPN = WireGuard/OpenVPN等真正的VPN协议）
    7.4.1 VPN连接时发生了什么（完整步骤）
        Step 1: 与VPN服务器建立加密隧道
        Step 2: 系统创建虚拟网卡（tun0 / utun3）
        Step 3: 从VPN服务器获得一个新IP（如10.8.0.2）
        Step 4: 路由表被修改——添加规则，让流量走虚拟网卡
        Step 5: DNS服务器设置被修改——改为VPN服务器提供的DNS

    7.4.2 【关键】连VPN前后的路由表对比

        - 标注哪些行是新增的，哪些是被覆盖的
        - 关键变化：default路由从 en0(Wi-Fi) 变成 utun3(VPN)

    7.4.3 VPN全局模式（Full Tunnel）
        - 所有流量走VPN = 默认路由指向VPN虚拟网卡
        - DNS也走VPN = DNS服务器被改为VPN端的DNS
        - 结果：你"搬"到了VPN服务器所在的网络

    7.4.4 VPN分流模式（Split Tunnel）
        - 只有特定网段走VPN，其他走原来的路
        - WireGuard的AllowedIPs机制：只给指定网段添加路由
        - DNS可能走VPN也可能不走——取决于配置
        - 这就是企业VPN的典型模式

    7.4.5 【关键】为什么开了真VPN后公司内网域名打不开？
        - 场景还原：你在公司，连着公司Wi-Fi（DNS=公司DNS服务器）
        - 你开了个人VPN（WireGuard/OpenVPN，全局模式）
        - VPN把DNS改成了VPN服务器端的DNS（如8.8.8.8）
        - 公司内部域名（如 git.internal.company.com）只有公司DNS才能解析
        - 结果：DNS查询发到8.8.8.8，它不认识公司域名 → 解析失败！
        - 解决方案：Split Tunnel + DNS分流
        - ⚠️ 如果你用的不是真VPN，而是Clash/V2Ray等代理工具 → 看§7.6.2

7.5 代理的本质：应用层转发，不改路由表（这里的代理 = Clash/V2Ray/Shadowsocks等）
    7.5.0 正向代理 vs 反向代理（简要辨析）
        - 正向代理 = 帮客户端转发（本章重点，属于流量控制）
        - 反向代理 = 帮服务端分发（不属于本章范畴，详见第10章自建服务）
        - 本章只讨论正向代理及其变体（系统代理/TUN模式）
    7.5.1 系统代理模式
        - 在系统设置中配置HTTP/SOCKS5代理
        - 只有"尊重系统代理设置"的应用才走代理
        - 路由表完全不变！
        - DNS查询通常不走代理（DNS泄漏问题）

    7.5.2 应用级代理
        - 单独为某个应用配置代理（浏览器/终端/Git/SSH）
        - 更精细的控制，但更麻烦
        - 每个应用的配置方式不同

    7.5.2.1 HTTP代理 vs SOCKS5代理——有什么区别？
        - HTTP代理（端口7890）：只能代理HTTP/HTTPS流量
        - SOCKS5代理（端口7891）：可以代理任意TCP/UDP流量（更通用）
        - 为什么终端常用ALL_PROXY=socks5://... 而不是HTTP代理？
          因为很多命令行工具走的不是纯HTTP协议（如SSH、数据库连接）
        - 实际选择：浏览器用HTTP代理够了，终端建议用SOCKS5

    7.5.3 DNS泄漏：系统代理的致命弱点
        - 代理只转发HTTP/HTTPS请求
        - DNS查询走的是系统的UDP 53端口 → 不经过代理 → 直接发给ISP的DNS
        - ISP的DNS被污染 → 返回错误IP → 即使有代理也访问不了
        - 这就是为什么"系统代理"模式下有些网站打不开

    7.5.4 【关键】Clash的模式体系——两个维度的组合
        ⚠️ 很多人分不清Clash的各种"模式"，因为Clash实际上有两个独立的维度：

        第一维度：流量怎么拦截（捕获方式）——决定"哪些流量被Clash接管"
          | 捕获方式 | 做了什么 | 被接管的范围 |
          |---------|---------|-------------|
          | 系统代理 | 设置HTTP/SOCKS代理(127.0.0.1:7890) | 只有尊重系统代理的应用（浏览器等），终端/部分App不走 |
          | TUN模式 | 创建虚拟网卡+修改路由表 | 所有应用的所有流量，无一例外 |

        第二维度：拦截到的流量怎么处理（路由模式）——决定"走代理还是直连"
          | 路由模式 | 做了什么 | 效果 |
          |---------|---------|------|
          | 全局(Global) | 所有被拦截流量 → 全部走代理服务器 | 国内国外都走VPS，国内变慢，全部消耗机场流量 |
          | 规则(Rule) | 按规则文件逐条判断 → PROXY或DIRECT | 国外走代理，国内直连，最聪明的模式 |
          | 直连(Direct) | 所有被拦截流量 → 全部直连 | 等于关闭代理，但Clash仍在运行 |

        两个维度自由组合，产生6种实际模式：
          | 捕获 × 路由 | 浏览器Google | 终端curl | 百度 | 公司内网 |
          |---|---|---|---|---|
          | 系统代理+全局 | 走代理✓ | 不走✗ | 走代理(慢) | 走代理(可能不通) |
          | 系统代理+规则 | 走代理✓ | 不走✗ | 直连(快) | 直连(看规则) |
          | 系统代理+直连 | 直连(被墙) | 直连 | 直连 | 直连 |
          | TUN+全局 | 走代理✓ | 走代理✓ | 走代理(慢) | 走代理(不通!) |
          | **TUN+规则** | **走代理✓** | **走代理✓** | **直连(快)** | **看bypass规则** |
          | TUN+直连 | 直连(被墙) | 直连 | 直连 | 直连 |

        ★ 日常推荐：**TUN + 规则模式**——全面拦截 + 智能分流

    7.5.5 【关键】为什么国内流量不消耗"机场流量"？
        "流量"指的是你购买的代理服务（机场）的带宽配额，不是你家宽带的流量。

        规则模式下的流量路径：
          访问Google（规则判断 → PROXY）：
            你的电脑 →[加密]→ 代理服务器(VPS) → Google → VPS →[加密]→ 你
            数据经过了VPS → VPS消耗带宽 → 计入机场流量

          访问百度（规则判断 → DIRECT）：
            你的电脑 → 百度 → 你
            数据完全没经过VPS → VPS零消耗 → 不计流量

        DIRECT规则的流量根本不经过代理服务器，跟没开Clash一模一样。

        Clash怎么判断"国内还是国外"？——规则引擎：
          - 域名规则：google.com → PROXY，baidu.com → DIRECT
          - GeoIP规则：目标IP属于中国（查GeoIP数据库）→ DIRECT
          - 私有IP规则：192.168.x.x / 10.x.x.x → DIRECT
          - 兜底规则：以上都不匹配 → PROXY（默认走代理更安全）
          （Clash规则文件示例在§7.8.3展示）

7.6 TUN模式：代理工具的"全局化"（Clash/Surge的TUN/增强模式）
    7.6.1 TUN模式做了什么
        - 创建虚拟网卡（类似VPN）
        - 修改路由表，让所有流量流入虚拟网卡
        - 虚拟网卡把流量交给代理客户端处理
        - 代理客户端根据规则决定：直连？还是走代理服务器？

    7.6.2 【关键场景】公司电脑开Clash TUN模式，内网网站打不开
        场景还原：
        - 你在公司电脑上开了Clash Verge（TUN模式）
        - 访问 google.com → 成功（走代理）
        - 访问 git.internal.company.com → 失败！

        失败的两层原因：
        第一层：DNS解析失败
          - Clash拦截了DNS查询
          - Clash把DNS请求发给代理服务器（境外VPS）去解析
          - 境外DNS根本不认识 git.internal.company.com（公司内网域名）
          - 返回NXDOMAIN → 域名不存在

        第二层：即使DNS成功，路由也不通
          - 假设DNS侥幸解析出公司内网IP 10.1.2.3
          - Clash判断规则：匹配默认规则PROXY → 发给代理服务器
          - 代理服务器在境外，尝试连接10.1.2.3 → 10.x.x.x是内网地址，到不了！
          - 连接超时

        Bypass为什么能修复？
          - 添加规则：*.company.com → DIRECT
          - Clash看到DIRECT → DNS走本地（公司DNS）→ 解析成功
          - 连接走本地网络（不经过代理）→ 直连10.1.2.3 → 成功
          - **DIRECT同时解决了DNS和路由两个问题**

    7.6.3 TUN模式 vs 真VPN的区别
        | 维度 | VPN | TUN模式 |
        |------|-----|---------|
        | 虚拟网卡 | 有 | 有 |
        | 修改路由表 | 是 | 是 |
        | 你的IP | 变了（获得VPN网络IP） | 没变（还是原来的IP） |
        | 流量处理 | 全部加密送到VPN服务器 | 代理客户端按规则分流 |
        | DNS | 接管（走VPN的DNS） | 接管（fake-ip或转发到代理DNS） |

    7.6.4 fake-ip模式
        - 代理客户端拦截DNS请求
        - 返回一个假IP（如198.18.x.x）给应用
        - 应用发起连接时，代理客户端根据域名（不是IP）决定走向
        - 好处：DNS永远不泄漏，域名级别的精确分流

7.7 【核心对比】各模式下的DNS和路由行为

    | 模式 | DNS查询走哪里 | 流量走哪里 | 内网域名 | 内网IP可达？ |
    |------|-------------|-----------|---------|------------|
    | 直连 | 系统DNS（公司/ISP） | 本地网络 | ✓ 能解析 | ✓ 能到达 |
    | 真VPN全局 | VPN的DNS | VPN隧道 | ✗ VPN DNS不认识 | ✗ 走VPN出去了 |
    | 真VPN分流 | 看配置 | 看配置 | 取决于DNS配置 | 取决于路由配置 |
    | 代理-系统代理 | 系统DNS（不走代理） | 匹配的走代理 | ✓ 能解析（DNS没变） | ✗ 如果走了代理，代理服务器到不了内网 |
    | 代理-TUN模式 | 代理客户端接管 | 全部经代理客户端分流 | ✗ 代理DNS不认识 | ✗ 代理服务器到不了内网 |
    | TUN+bypass | 内网域名走本地DNS | 内网走DIRECT | ✓ 公司DNS解析 | ✓ 本地直连 |

    关键认知：内网访问失败往往是**双重故障**——DNS解析失败 + 代理服务器路由不可达。
    bypass/DIRECT规则同时解决这两个问题。

7.8 分流与Bypass：鱼和熊掌兼得
    7.8.1 为什么需要分流？
        - 全局VPN/TUN：所有流量都走远端 → 内网服务不可用 + 国内网站变慢
        - 需求：国外走代理，国内直连，公司内网直连

    7.8.2 VPN的分流：AllowedIPs

        - 局限：只能按IP/网段分流，不能按域名

        - 按IP段分流（192.168.x.x直连，其他走代理）
        - 按地理位置分流（GeoIP：国外走代理，国内直连）

    7.8.4 【实战场景】公司VPN + 个人代理工具共存

7.9 GFW与翻墙原理

    7.9.1 墙怎么工作
        - DNS污染（返回错误IP）
        - IP封锁（直接封Google的IP段）
        - 深度包检测DPI（识别VPN/代理协议特征）

        - VLESS + Reality：最新的伪装技术

    7.9.4 真VPN vs 代理协议在翻墙中的选择
        - 真VPN协议（WireGuard/OpenVPN）：简单但协议特征明显，容易被墙识别和封锁
        - 代理协议（V2Ray/Trojan/VLESS）：灵活、可伪装成HTTPS流量，是目前主流
        - ⚠️ 日常所说的"VPN翻墙"，绝大多数实际使用的是代理协议（V2Ray/Trojan），不是真正的VPN协议

7.10 【完整场景演示】三种模式访问Google的数据包之旅

    场景一：直连（被墙）
    ```
    DNS查询: 你 → ISP DNS → 被DNS污染，返回错误IP
    或者: 你 → Google IP → 被IP封锁，连接超时
    结果：访问失败
    ```

    场景二：真VPN全局模式（WireGuard/OpenVPN）
    ```
    DNS查询: 你 → tun0(VPN) → 加密隧道 → VPN服务器 → 8.8.8.8 → 返回正确IP
    TCP连接: 你 → tun0(VPN) → 加密隧道 → VPN服务器 → Google → 返回网页
    路由表: default → utun3 (VPN虚拟网卡)
    结果：成功，但所有流量都走VPN（国内网站也变慢）
    ```

    场景三：代理TUN模式（Clash）
    ```
    DNS查询: 你 → tun0(Clash) → Clash判断：google.com匹配代理规则
            → 通过代理服务器查DNS → 返回正确IP
    TCP连接: 你 → tun0(Clash) → Clash判断：走代理
            → 加密 → 代理服务器 → Google → 返回网页
    国内网站: 你 → tun0(Clash) → Clash判断：baidu.com匹配直连规则
            → 直接连接百度 → 返回网页
    结果：成功，Google走代理，百度直连，两不误
    ```

    三者对比总结表：
    | 维度 | 直连 | 真VPN全局（WireGuard） | 代理TUN（Clash） |
    |------|------|---------|---------|
    | 路由表 | 默认→网关 | 默认→VPN网卡 | 默认→Clash网卡 |
    | DNS | 系统DNS | VPN的DNS | Clash接管 |
    | Google | ✗ 被墙 | ✓ 走VPN | ✓ 走代理 |
    | 百度 | ✓ 直连 | ✓ 但走VPN（慢） | ✓ 直连（快） |
    | 公司内网 | ✓ | ✗ DNS失败 | 取决于bypass规则 |

7.11 本章小结
    - 日常说的"VPN"很多时候其实是代理工具——搞清楚自己用的是哪个
    - 真VPN的本质 = 虚拟网卡 + 修改路由表 + 接管DNS（WireGuard/OpenVPN）
