## 0.1 为什么需要一张全局地图？

学网络最怕的就是：**每个概念都听过，但串不起来。**

DNS、IP、路由器、防火墙、NAT……单独看都懂，一出问题就懵。原因很简单——你脑子里没有一条**完整的数据流动路径**。

这章就干一件事：画出这条路径。

**为什么选 Google 作为案例？**

访问 google.com 天然地涵盖了网络知识中最精华的部分：
- 需要 DNS 解析（域名→IP）
- 需要 TCP 建立连接
- 需要 TLS 加密（HTTPS）
- 需要 NAT 转换（内网→公网）
- **在中国直连访问不了**（引出第 7 章：GFW、VPN、代理）

## 0.2 缩写速查表

> 遇到不认识的缩写来这里查，现在不需要背完。详细解释见各章。

| 缩写 | 全称 | 一句话解释 |
|-----|-----|---------|
| **IP** | Internet Protocol | 设备在网络中的"门牌号"，如 `192.168.1.100` |
| **MAC** | Media Access Control | 网卡的"身份证号"，出厂时烧录，全球唯一 |
| **DNS** | Domain Name System | 把域名翻译成 IP 地址的"电话簿" |
| **DHCP** | Dynamic Host Configuration Protocol | 自动给设备分配 IP 地址 |
| **ARP** | Address Resolution Protocol | 在局域网内通过 IP 找 MAC 地址 |
| **NAT** | Network Address Translation | 内网 IP 换公网 IP（路由器的核心功能之一）|
| **TCP** | Transmission Control Protocol | 可靠传输，保证数据完整送达（"挂号信"）|
| **UDP** | User Datagram Protocol | 快速传输，不保证送达（"明信片"）|
| **HTTP** | HyperText Transfer Protocol | 浏览器和网站之间的通信协议（明文）|
| **HTTPS** | HTTP Secure | HTTP 的加密版，地址栏有 🔒 |
| **TLS** | Transport Layer Security | HTTPS 底层的加密机制 |
| **SSH** | Secure Shell | 远程登录服务器的加密通道 |
| **VPN** | Virtual Private Network | 创建虚拟网卡+修改路由表，建立加密隧道 |
| **ISP** | Internet Service Provider | 运营商（电信/联通/移动）|
| **CDN** | Content Delivery Network | 把内容分发到全国各地，让你访问最近的节点 |
| **GFW** | Great Firewall | 防火长城，拦截境外部分网站 |
| **TUN** | TUN device | 操作系统的虚拟网络接口，VPN/代理工具常用 |

## 0.3 场景设定：你坐在家里，打开笔记本，要访问 google.com

### 家庭网络拓扑图

```
你的笔记本
192.168.1.100
     │
     │ Wi-Fi（无线）
     ▼
┌─────────────────┐
│     路由器       │  ← 分配 IP（DHCP）、做 NAT、当防火墙
│  192.168.1.1    │     LAN 口：192.168.1.1
│  公网 IP: 1.2.3.4│     WAN 口：1.2.3.4（运营商分配的公网 IP）
└─────────────────┘
     │
     │ 网线
     ▼
┌─────────────────┐
│     光猫         │  ← 把光信号翻译成电信号（光电转换）
└─────────────────┘
     │
     │ 光纤
     ▼
┌─────────────────┐
│   运营商网络      │  ← 电信/联通/移动，负责把你的数据发到互联网
└─────────────────┘
     │
     ▼
┌─────────────────┐
│    互联网         │  ← 全球互联的网络
└─────────────────┘
     │
     ▼
┌─────────────────┐
│  Google 服务器   │  ← 142.250.x.x（在香港/美国等境外）
└─────────────────┘
```

**你的笔记本状态（连上 Wi-Fi 后）：**
- IP 地址：`192.168.1.100`（路由器通过 DHCP 分配）
- 默认网关：`192.168.1.1`（路由器的 IP）
- DNS 服务器：`192.168.1.1`（或运营商 DNS）
- MAC 地址：`aa:bb:cc:dd:ee:ff`（出厂烧录，唯一）

## 0.4 🔑 场景一：直连访问 google.com 的完整流程

### Step 0：你的电脑怎么有 IP 地址的？（DHCP）

你一连上 Wi-Fi，在任何网络操作之前，DHCP 已经悄悄发生了：

```
笔记本：（广播）嗨，我新来的，谁能给我分配一个 IP？
路由器：好，给你 192.168.1.100，子网掩码 255.255.255.0，网关 192.168.1.1，DNS 192.168.1.1
笔记本：收到，我以后就用这个 IP 了。
```

> 📖 DHCP 详细原理 → 第 3 章 §3.2

---

### Step 1：浏览器说"我要访问 google.com"——DNS 查询

浏览器只认识 IP，不认识域名。所以第一步是**把 google.com 翻译成 IP**。

**DNS 查询顺序：**

```
1. 浏览器缓存：我以前查过 google.com 吗？——没有
2. 操作系统缓存：DNS 缓存里有吗？——没有
3. hosts 文件：/etc/hosts 里有 google.com 的记录吗？——没有
4. 向 DNS 服务器查询：问 192.168.1.1（路由器/DNS 服务器）
```

**DNS 查询本质上是一个 UDP 数据包：**

```
┌─────────────────────────────────────────────────────────────┐
│                      以太网帧                                  │
│  [目标 MAC: 路由器 MAC] [源 MAC: 笔记本 MAC]                   │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                      IP 包                               │  │
│  │  [源 IP: 192.168.1.100] [目标 IP: 192.168.1.1]         │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │                    UDP 包                          │  │  │
│  │  │  [源端口: 随机(如52341)] [目标端口: 53]             │  │  │
│  │  │  ┌───────────────────────────────────────────┐  │  │  │
│  │  │  │           DNS 查询内容                        │  │  │  │
│  │  │  │  "google.com 的 A 记录是什么？"               │  │  │  │
│  │  │  └───────────────────────────────────────────┘  │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**这个数据包怎么发出去？**

发数据包之前，笔记本必须知道"下一跳"（路由器）的 MAC 地址——这就是 ARP 的作用：

```
笔记本查路由表：192.168.1.1 在局域网段 → 直接送达
但以太网传输需要 MAC 地址，不能只用 IP！

ARP 广播：（对局域网所有设备喊）谁是 192.168.1.1？告诉我你的 MAC！
路由器：是我！我的 MAC 是 11:22:33:44:55:66
笔记本：记住了，以后发给 192.168.1.1 就用 MAC 11:22:33:44:55:66
```

**以太网帧最终构造：**

```
[目标 MAC: 11:22:33:44:55:66（路由器）]
[源 MAC: aa:bb:cc:dd:ee:ff（笔记本）]
[IP 包内容...]
```

**路由器收到 DNS 查询包后：**
1. 检查目标 IP（192.168.1.1）= 自己 → 这是问我的 DNS 查询
2. 路由器自己查（或转发给运营商 DNS）
3. 最终结果：`google.com → 142.250.80.46`（假设这是 Google 的 IP）
4. 把结果通过 UDP 回复给笔记本

> 📖 DNS 详细原理 → 第 3 章 §3.4  
> 📖 ARP 详细原理 → 第 2 章 §2.6  
> ⚠️ **重要认知**：DNS 查询是普通 UDP 数据包，同样要查路由表。如果路由表被 VPN 修改，DNS 查询也走 VPN！（详见第 7 章）

---

### Step 2：查路由表，决定数据往哪走

笔记本路由表（简化版）：

```
Destination    Gateway        Netif
default        192.168.1.1    en0    ← 不认识的目标，发给路由器
192.168.1.0/24 link#7         en0    ← 局域网，直接送达
127.0.0.1      127.0.0.1      lo0    ← 本机
```

**笔记本想连接 142.250.80.46（Google）：**
- 142.250.80.46 不在局域网段（192.168.1.0/24）
- 不是本机地址（127.x.x.x）
- 匹配 `default` → 发给网关 192.168.1.1

> 📖 路由表详细原理 → 第 4 章 §4.4

---

### Step 3：建立 TCP 连接（三次握手）

知道了 Google 的 IP，开始建立连接。目标端口：443（HTTPS）。

**三次握手过程：**

```
笔记本（192.168.1.100:52341）                   Google（142.250.80.46:443）
      │                                              │
      │ ──── SYN（Seq=0，"我想连你"）───────────────→ │
      │                                              │
      │ ←─── SYN-ACK（Seq=0, Ack=1，"好，我准备好了"）│
      │                                              │
      │ ──── ACK（Ack=1，"收到，开始通信"）──────────→ │
      │                                              │
      │             连接建立完成                        │
```

**但这个 SYN 包是怎么从家里到 Google 的？**

路由器在中间做了 **NAT（网络地址转换）**：

```
笔记本发出：
  源 IP: 192.168.1.100   目标 IP: 142.250.80.46
  源端口: 52341          目标端口: 443

路由器 NAT 转换后发出：
  源 IP: 1.2.3.4（公网IP）目标 IP: 142.250.80.46
  源端口: 63001（重新分配）目标端口: 443

路由器 NAT 表记录：
  内网 192.168.1.100:52341 ↔ 公网 1.2.3.4:63001
  （用于把返回的包重新转发给笔记本）
```

**Google 看到的**：请求来自 `1.2.3.4`（你家路由器的公网 IP），不知道你家内网有多少设备。

> 📖 TCP 三次握手 → 第 4 章 §4.3  
> 📖 NAT 详细原理 → 第 5 章 §5.2

---

### Step 4：TLS 握手（建立加密通道）

TCP 连接建立后，HTTPS 还需要 TLS 握手，建立加密通道：

```
简化过程：
笔记本：你好，我支持这些加密算法：AES-256、ChaCha20...
Google：好，我们用 AES-256。这是我的证书（证明我就是 google.com）
笔记本：（验证证书）证书有效，CA 可信。好，我们用这个密钥通信。
双方：加密通道建立，以后所有数据都加密传输
```

**为什么需要 TLS？**

```
没有 TLS（HTTP）：
  你发出：GET / HTTP/1.1 Host: google.com
  路由器/运营商/中间人 都能看到这段明文

有了 TLS（HTTPS）：
  你发出：[加密乱码，只有 Google 能解密]
  路由器/运营商 只知道你在和 142.250.80.46 通信，不知道内容
```

> 📖 TLS/HTTPS 详细原理 → 第 6 章 §6.4

---

### Step 5：发送 HTTP 请求

```
GET / HTTP/1.1
Host: www.google.com
User-Agent: Mozilla/5.0 (Mac)
Accept: text/html
Connection: keep-alive
```

这段内容被 TLS 加密后，经过：
```
笔记本 → Wi-Fi → 路由器（NAT）→ 光猫 → 运营商 → 骨干网 → Google 服务器
```

---

### Step 6：响应返回，逐层拆包

Google 服务器返回 `200 OK` + HTML 内容，数据包沿原路返回：

```
Google → 运营商 → 路由器（NAT反向转换）→ 笔记本

路由器 NAT 反查：
  收到目标端口 63001 的包
  查 NAT 表：63001 ↔ 192.168.1.100:52341
  把目标 IP 改回 192.168.1.100，转发给笔记本
```

笔记本收到数据包，逐层拆包：

```
以太网帧 → 取出 IP 包
IP 包 → 取出 TCP 段
TCP 段 → 取出 TLS 加密数据
TLS 解密 → 取出 HTTP 响应
HTTP 解析 → 取出 HTML 内容
浏览器渲染 → 你看到 Google 首页
```

---

### Step 7：完整数据流向图

```
┌──────────────────────────────────────────────────────────────┐
│                       完整数据包之旅                             │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  笔记本                路由器              运营商        Google │
│  [192.168.1.100]      [1.2.3.4]                              │
│                                                               │
│  1. DHCP ────────────→ 分配 IP                                │
│                                                               │
│  2. DNS 查询(UDP:53) → 转发/解析 → 返回 142.250.80.46          │
│     [ARP 获取路由器 MAC，构造以太网帧]                            │
│                                                               │
│  3. TCP SYN ─────────→ NAT 转换 ──────────────────→ Google   │
│     (52341→443)         (63001→443)                          │
│                                                               │
│  4. TLS 握手（加密密钥协商，验证 Google 证书）                     │
│                                                               │
│  5. HTTP GET / ───────→ NAT ──────→ 运营商 ──────→ Google     │
│     [TLS 加密后，对中间节点不可见]                                │
│                                                               │
│  6. 200 OK + HTML ←── NAT 反转 ←── 运营商 ←──────── Google    │
│                                                               │
│  7. 浏览器渲染 → 你看到 Google 首页                              │
│                                                               │
│  ★ 整个过程不到 1 秒                                             │
└──────────────────────────────────────────────────────────────┘
```

> ⚠️ **在中国，直连访问 google.com 会在 Step 2（DNS 被污染）或 Step 3（IP 被封锁）失败。**  
> 这就引出了第 7 章的核心问题：VPN 和代理工具怎么绕过这个问题？

## 0.5 场景二预览：开了真 VPN（WireGuard/OpenVPN）后，流程怎么变？

> 这是高层预览，完整原理见第 7 章 §7.4。

**VPN 连接后发生了什么：**

```
系统新增：
  - 虚拟网卡 utun3，IP: 10.8.0.2（VPN 分配）
  - 路由表被修改：所有流量的默认下一跳从"路由器"变成"VPN 虚拟网卡"
  - DNS 服务器被改成 VPN 服务器端的 DNS（如 8.8.8.8）
```

**数据流向的变化：**

```
没有 VPN：
  笔记本 → 路由器 → 运营商 → [GFW 拦截] → ×Google

有了 VPN（全局模式）：
  笔记本 → VPN 虚拟网卡 → 加密隧道 → VPN 服务器（境外）→ Google
                              ↑
               GFW 只看到你在和 VPN 服务器通信（加密内容），不知道你要访问 Google
```

**DNS 变化（关键！）：**

```
没有 VPN：DNS 查询 → 路由器/ISP DNS
有了 VPN：DNS 查询 → 也走 VPN 隧道 → VPN 服务器端的 DNS（如 8.8.8.8）

影响：如果你同时在公司网络，公司内网域名（git.company.com）
      用 8.8.8.8 查不到 → 公司内网打不开！
      → 详见第 7 章 §7.4.5
```

## 0.6 场景三预览：开了代理工具（Clash 等）后，流程怎么变？

> 这是高层预览，完整原理见第 7 章 §7.5-7.6。

> ⚠️ **注意：日常说的"开VPN翻墙"，很多时候其实是开的代理工具（Clash/V2Ray 等）——它们不是真 VPN，区别很大，详见第 7 章 §7.2。**

### 系统代理模式（Clash 默认）

```
变化：系统设置里多了一条代理配置（127.0.0.1:7890）
路由表：完全没变！

浏览器访问 google.com：
  浏览器 → 127.0.0.1:7890（Clash）→ 加密 → VPS → Google ✅

终端 curl google.com：
  curl → 路由表 → 路由器 → [GFW 拦截] → × ❌
  （curl 不看系统代理设置，直连被墙）
```

### TUN 模式（增强模式）

```
变化：创建虚拟网卡，路由表修改（类似 VPN）

笔记本所有流量 → Clash 虚拟网卡 → Clash 按规则决定：
  google.com → 规则：PROXY → 通过代理服务器访问 ✅
  baidu.com  → 规则：DIRECT → 直连百度 ✅
  git.company.com → 如果没有 bypass 规则 → PROXY → 代理服务器到不了内网 ❌
                    如果有 bypass 规则（DIRECT）→ 本地 DNS + 直连 ✅
```

**直连 vs VPN vs 代理工具 的核心对比：**

| | 直连 | 真 VPN（全局）| 代理工具 TUN |
|--|-----|------------|------------|
| 路由表 | 不变 | 改变（走 VPN）| 改变（走 Clash）|
| DNS | 本地/ISP | VPN 服务器 DNS | Clash 接管 |
| 访问 Google | ❌ 被墙 | ✅ | ✅ |
| 公司内网 | ✅ | ❌（DNS 失败）| 看 bypass 规则 |

> 📖 完整原理 → 第 7 章（全书最重要章节）

## 0.7 极简网络分层

网络工程师把数据包的旅程分成了几层，每层管一件事：

```
┌────────────────────────────────────────────────────────┐
│  应用层    │ 说什么？  │ HTTP、HTTPS、DNS、SSH       │→ 第 4 章、第 6 章
├────────────────────────────────────────────────────────┤
│  传输层    │ 怎么送？  │ TCP（可靠）、UDP（快速）    │→ 第 4 章
├────────────────────────────────────────────────────────┤
│  网络层    │ 送到哪？  │ IP 地址、路由表            │→ 第 2 章、第 4 章
├────────────────────────────────────────────────────────┤
│  链路/物理层│ 用什么送？│ 以太网帧、Wi-Fi 帧、光纤    │→ 第 1 章、第 2 章
└────────────────────────────────────────────────────────┘
```

**对应到 Step 0.4 的完整流程：**

```
应用层：浏览器生成 HTTP 请求
传输层：TCP 三次握手，建立可靠连接
网络层：查路由表，选择下一跳；NAT 转换 IP
链路层：ARP 获取 MAC，构造以太网帧，在局域网内传输
```

> 🔑 你不需要死记七层 OSI 模型，日常工作只需要理解这四层。

## 0.9 🔬 动手验证：traceroute 看数据经过哪些节点

```bash
# Mac / Linux
traceroute google.com

# 或者用更好用的 mtr（需要安装）
mtr google.com

# Windows
tracert google.com
```

**输出示例（在国内，访问 Google 大概率看到星号）：**

```
traceroute to google.com (142.250.80.46), 30 hops max
 1  192.168.1.1       1.2 ms     ← 你的路由器
 2  10.x.x.x          5.1 ms     ← 光猫/运营商
 3  61.152.x.x        8.3 ms     ← 运营商骨干网
 4  202.97.x.x       15.6 ms     ← 运营商国际出口
 5  * * *                         ← 到 Google 的路被墙了，没有响应
 6  * * *
...（超时，连接不上）
```

每一行 = 数据包经过的一个"跳"（路由器/节点）。星号 `* * *` = 这一跳没有响应（可能是路由器禁 ICMP，也可能是被 GFW 阻断）。

## 本章小结

| 概念 | 一句话 | 详见 |
|-----|-------|------|
| DHCP | 路由器自动给你分配 IP | 第 3 章 |
| ARP | 通过 IP 查 MAC，构造以太网帧 | 第 2 章 |
| DNS | 把域名翻译成 IP（UDP:53 的数据包）| 第 3 章 |
| 路由表 | 每台电脑都有，决定数据包从哪个网卡出去 | 第 4 章 |
| TCP 三次握手 | SYN→SYN-ACK→ACK，建立可靠连接 | 第 4 章 |
| NAT | 路由器把内网 IP 换成公网 IP | 第 5 章 |
| TLS | HTTPS 的加密机制，防止中间人偷看 | 第 6 章 |
| GFW | 在 DNS 和 IP 层面封锁境外网站 | 第 7 章 |
| VPN/代理 | 绕过 GFW 的两种技术路线 | 第 7 章 ⭐ |

**🔑 核心认知：** 网络不神秘。数据包从 A 到 B 的每一步都有迹可循：DHCP 拿 IP → ARP 找 MAC → DNS 查地址 → 路由表选路 → TCP 建连接 → NAT 转地址 → TLS 加密 → 到达目标。

## 📦 扩展区域

> 以下内容第一遍可跳过。

### OSI 七层模型（面试可能问）

我们简化成了 4 层，完整的 OSI 模型有 7 层：

| OSI 层 | 名称 | 对应简化层 | 常见协议 |
|-------|------|----------|---------|
| 7 | 应用层 | 应用层 | HTTP/HTTPS/DNS/SSH |
| 6 | 表示层 | 应用层 | TLS/SSL（加密在这层）|
| 5 | 会话层 | 应用层 | — |
| 4 | 传输层 | 传输层 | TCP/UDP |
| 3 | 网络层 | 网络层 | IP/ICMP/ARP |
| 2 | 数据链路层 | 链路层 | 以太网/Wi-Fi |
| 1 | 物理层 | 物理层 | 网线/光纤/无线电 |

> 日常工作没人说"第 6 层有问题"，都说"DNS 挂了"、"443 端口不通"这种具体描述。不用死记 7 层。

### 数据封装：从 HTTP 请求到以太网帧

数据从应用层往下，每过一层都被"套一层信封"：

```
HTTP 请求（你要发的内容）
    ↓ 传输层套上 TCP 头
[TCP 头（源端口/目标端口/序列号）| HTTP 请求]
    ↓ 网络层套上 IP 头
[IP 头（源 IP/目标 IP）| TCP 头 | HTTP 请求]
    ↓ 链路层套上以太网帧头
[以太网头（源 MAC/目标 MAC）| IP 头 | TCP 头 | HTTP 请求]
    ↓ 物理层
在网线/Wi-Fi 上传输的电信号/无线电信号
```

收到数据时逆向拆包，每层只看自己那一层的头。
