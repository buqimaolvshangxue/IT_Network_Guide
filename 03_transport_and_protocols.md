# 第3章：传输与协议 - 数据怎么送、送到哪

## 这章解决什么问题

前面两章我们搞清楚了"网络是什么"和"数据怎么找到目标机器"。但现在有个新问题：

- 你的电脑上同时开着浏览器、微信、SSH终端、数据库，**数据包到了你的电脑，怎么知道该给谁？**
- 发邮件和看视频，一个要求"必须送到"，一个要求"快点就行"，**能不能用不同的送货方式？**
- 浏览器输入 `https://example.com/login`，这一串字符**电脑怎么理解的？**

这章就解决这些问题：**端口号区分服务、传输协议选择送货方式、应用层协议定义说话规则**。

## 🚦 快速通关模式（先读这个）

### 30 秒先懂

- **IP 地址**决定"送到哪台机器"
- **端口号**决定"送给这台机器上的哪个程序"
- **TCP / UDP**决定"怎么送"（可靠优先还是速度优先）
- **HTTP / SSH / DNS**决定"送过去后说什么话"

你只要先把这 4 句话串起来，后面细节就不容易乱。

### 最小可用（先会用，再深挖）

1. 先记住常用端口：`22(SSH)`、`80(HTTP)`、`443(HTTPS)`。
2. 先会判断：要可靠（登录/下单/传文件）用 TCP；要实时（语音/直播/游戏）优先 UDP。
3. 先会排查：用 `lsof`/`netstat` 看端口有没有监听，用 `nc -zv IP 端口` 看端口能不能通。

### 进阶再看（不着急）

- 先读 `1 端口号` + `2 TCP vs UDP`，够解决 80% 问题
- 再读 `3 常见应用层协议`，理解 HTTP/SSH/DNS 的差异
- 最后读 `4 路由概念` 和 `6 动手验证`，补全链路和实战

## 1. 端口号：区分同一台电脑上的不同程序

### 🔑 第一步：理解问题

**IP 地址能找到电脑，但找不到程序。**

你的电脑上同时有很多程序在联网：
- 浏览器在访问百度
- 微信在收消息
- 邮件客户端在同步

数据包到了你的电脑，**操作系统怎么知道该给哪个程序？**

答案：用**端口号**区分。

---

### 🔑 第二步：什么是端口

**端口就是一个数字（0-65535），用来标识电脑上的不同程序。**

```
IP 地址 = 找到哪台电脑
端口号 = 找到电脑上的哪个程序
```

类比：
- IP 地址 = 公司地址（找到公司）
- 端口号 = 部门号码（找到具体部门）

---

### 🔑 第三步：一个连接需要四个数字

网络连接是**双向的**，双方都需要知道对方的地址：

```
你的电脑                              百度服务器
IP: 192.168.1.100                    IP: 220.181.38.148
端口: 52341                           端口: 443
         ↘                        ↙
            这四个数字确定一个连接
```

---

### 🔑 第四步：目标端口 vs 源端口

每个连接有两个端口，作用不同：

#### 目标端口（你要连的）

```
你的电脑 ──────→ 百度:443
                   ↑
              目标端口
```

- **是谁决定的**：服务端（百度决定用443）
- **作用**：你要访问的服务在哪个端口"站岗"
- **特点**：固定的、已知的

#### 源端口（标识你是谁）

```
你的电脑:52341 ──────→ 百度
       ↑
    源端口
```

- **是谁决定的**：操作系统自动分配
- **作用**：标识"这个连接是哪个程序的"
- **特点**：临时的、随机的

---

### 🔑 第五步：源端口的工作机制

**问题**：你开了3个浏览器标签页，都在访问百度，百度回复时怎么知道该给哪个标签页？

**答案**：每个标签页用不同的源端口。

```
你的电脑                              百度服务器
┌─────────────┐
│ 标签页1      │──源端口52341──→ 百度443
│ 标签页2      │──源端口52342──→ 百度443
│ 标签页3      │──源端口52343──→ 百度443
└─────────────┘
```

**完整流程**：

```
【发起连接】
标签页1 要连百度
    ↓
操作系统：我给你分配源端口52341
操作系统：记录下来 52341 = 标签页1

【发送请求】
数据包里写上：源端口52341，目标端口443
    ↓
发给百度

【百度回复】
百度看到：请求来自52341
百度回复：发回给52341

【接收回复】
操作系统收到数据，目标端口是52341
    ↓
操作系统查表：52341 = 标签页1
    ↓
把数据给标签页1

【连接关闭】
标签页1关闭 → 52341释放 → 可以分配给其他连接
```

**类比：公司前台**

| | 网络 | 公司前台 |
|--|------|--------|
| 3个标签页 | 3个员工给同一家客户打电话 |
| 源端口 | 员工的工位号 |
| 目标端口 | 客户的电话号码 |

员工A（工位101）打给客户，客户回电说"找101的"，前台就知道转给员工A。

---

### 🔑 第六步：端口是谁决定的？

| | 服务端端口 | 客户端端口（源端口） |
|--|----------|------------------|
| 谁决定 | 服务端程序 | 操作系统 |
| 能改吗 | 能，但要客户端配合 | 不用管，自动的 |
| 例子 | 百度用443，你不能改 | 你的浏览器用什么端口，不用管 |

**关键理解**：

> 服务端用什么端口，客户端就得连什么端口。
>
> 你只能改**自己的服务**，不能改别人的。

- 百度用443 → 你连443
- 你的SSH改成2222 → 别人连你要用2222

---

### 📋 常见端口号

| 端口号 | 服务 | 说明 |
|--------|------|------|
| **22** | SSH | 远程登录服务器 |
| **80** | HTTP | 网页浏览（不加密） |
| **443** | HTTPS | 网页浏览（加密） |
| **3306** | MySQL | 数据库 |
| **6379** | Redis | 缓存 |
| **8080** | HTTP备用 | 开发测试常用 |

---

### 📋 端口的三个范围

| 范围 | 名称 | 谁用 | 说明 |
|------|------|------|------|
| **0-1023** | 知名端口 | 系统服务 | 需要管理员权限 |
| **1024-49151** | 注册端口 | 第三方软件 | 普通用户可用 |
| **49152-65535** | 动态端口 | 操作系统分配 | 源端口从这里挑 |

---

### 🤔 常见问题

**Q1：端口是所有程序共享的吗？**

是的，0-65535是所有程序共享的池子。同一个端口同一时间只能被一个程序使用。

**Q2：怎么改服务端端口？**

在服务端配置文件里改：

| 服务 | 配置文件 | 改哪里 |
|-----|---------|-------|
| SSH | `/etc/ssh/sshd_config` | `Port 22` → `Port 2222` |
| Nginx | `/etc/nginx/nginx.conf` | `listen 80` → `listen 8080` |

改完后重启服务，客户端连接时要指定新端口。

**Q3：客户端连接时怎么指定端口？**

```bash
# SSH 指定端口
ssh -p 2222 user@server

# 网站指定端口
http://example.com:8080
```

---

### 💻 查看端口的命令

```bash
# 查看哪些端口在监听
sudo lsof -iTCP -sTCP:LISTEN -n -P

# 查看特定端口被谁占用
sudo lsof -i :8080

# 测试端口能不能连
nc -zv example.com 443
```

## 2. TCP vs UDP：两种送货方式

### 🔑 核心区别

网络传输有两种主流协议，就像快递有两种送货方式：

| 特性 | TCP（Transmission Control Protocol，传输控制协议） | UDP（User Datagram Protocol，用户数据报协议） |
|------|---------------------|----------------------|
| **类比** | 📞 打电话 | 📱 发短信 |
| **连接** | 需要先"握手"建立连接 | 直接发，不管对方在不在 |
| **可靠性** | 保证送达、保证顺序 | 可能丢失、可能乱序 |
| **速度** | 慢一些（有确认机制） | 快（没有额外开销） |
| **应用场景** | 网页、文件传输、邮件 | 视频直播、游戏、DNS查询 |

### 📞 TCP：像打电话

**工作流程（三次握手）：**
1. **客户端**："喂，你在吗？"（SYN —— Synchronize，同步/请求连接）
2. **服务器**："在的，收到你的请求了"（SYN-ACK —— 同步确认）
3. **客户端**："好的，开始说话吧"（ACK —— Acknowledge，确认）

**特点：**
- ✅ 数据保证送达（丢了会重发）
- ✅ 数据保证顺序（乱了会重排）
- ✅ 有流量控制（不会把对方淹没）
- ❌ 速度慢一些（每次都要确认）

**日常例子：**
- 下载文件：必须完整，不能少一块
- 网页浏览：HTML代码必须完整
- 发邮件：不能丢字

### 📱 UDP：像发短信

**工作流程：**
1. **客户端**："这是消息，发了！"（直接发）
2. **服务器**：收到就收到，没收到也不知道

**特点：**
- ✅ 速度快（没有握手、确认）
- ✅ 开销小（数据包头部简单）
- ❌ 可能丢失（网络拥堵时）
- ❌ 可能乱序（走不同路径）

**日常例子：**
- 视频直播：丢几帧无所谓，流畅最重要
- 在线游戏：延迟低最重要，丢一个位置更新没事
- DNS查询：查不到就再查一次

### 📊 日常影响对比表

| 场景 | 协议 | 为什么 | 影响 |
|------|------|--------|------|
| 看网页 | TCP | 必须完整加载HTML/CSS/JS | 慢网络下会"转圈" |
| 看直播 | UDP | 丢几帧无所谓 | 卡顿时会"马赛克" |
| 下载文件 | TCP | 不能少一个字节 | 断网会暂停，恢复后继续 |
| 视频通话 | UDP | 实时性最重要 | 网络差时会"卡顿"但不会停 |
| 发邮件 | TCP | 必须完整送达 | 发送失败会有提示 |
| 玩FPS游戏 | UDP | 延迟低最重要 | 偶尔会"瞬移" |

## 3. 常见应用层协议：说什么话

传输层（TCP/UDP）解决了"怎么送"，应用层协议解决"说什么话"。

### 🌐 HTTP/HTTPS：网页浏览的语言

#### URL结构解析

```
https://www.example.com:443/path/to/page?id=123#section
│     │  │              │   │            │       │
│     │  │              │   │            │       └─ 锚点（页面内跳转）
│     │  │              │   │            └───────── 查询参数
│     │  │              │   └────────────────────── 路径
│     │  │              └────────────────────────── 端口（HTTPS默认443）
│     │  └───────────────────────────────────────── 域名
│     └──────────────────────────────────────────── 协议
```

**协议对比：**
- **HTTP**（80端口）：明文传输，像"明信片"，路由器能看到内容
- **HTTPS**（443端口）：加密传输，像"密封信件"，只有收件人能看

#### HTTP状态码速查

| 状态码 | 含义 | 生活类比 | 常见原因 |
|--------|------|----------|----------|
| **200** | 成功 | 快递送到了 | 正常访问 |
| **301** | 永久重定向 | 搬家了，新地址在这 | 网站换域名 |
| **302** | 临时重定向 | 临时去别处 | 短链接跳转 |
| **404** | 找不到 | 这个地址没人住 | 页面被删除/URL错误 |
| **500** | 服务器错误 | 收件人家里出事了 | 服务器代码bug |
| **502** | 网关错误 | 中转站出问题 | 后端服务挂了 |
| **503** | 服务不可用 | 收件人休假了 | 服务器过载 |

💻 **实际例子：**
```bash
# Mac/Linux - 查看HTTP响应
curl -I https://www.baidu.com

# 输出：
HTTP/2 200 
server: bfe
content-type: text/html
...
```

### 🔐 SSH：安全远程登录

**端口：22**

SSH（Secure Shell）是远程登录服务器的标准方式，所有数据都加密传输。

```bash
# Mac/Linux - SSH登录服务器
ssh username@192.168.1.100

# 指定端口（如果不是22）
ssh -p 2222 username@example.com

# 使用密钥登录
ssh -i ~/.ssh/id_ed25519 username@example.com  # 或 id_rsa
```

**日常用途：**
- 远程管理Linux服务器
- 通过跳板机访问内网
- 使用SSH隧道（端口转发）

### 📁 FTP/SFTP：文件传输

| 协议 | 端口 | 加密 | 说明 |
|------|------|------|------|
| FTP | 21 | ❌ | 古老协议，明文传输 |
| SFTP | 22 | ✅ | 基于SSH，安全 |

```bash
# Mac/Linux - SFTP传输文件
sftp username@example.com

# 进入SFTP后：
put local_file.txt          # 上传
get remote_file.txt         # 下载
ls                          # 列出远程文件
exit                        # 退出
```

## 4. 路由概念：数据包的"问路"过程

### 🔑 路由表：每台设备的"地图"

每台设备（电脑、路由器）都有一张**路由表**，记录"去某个网络，应该走哪个门"。

**查看路由表：**
```bash
# Mac/Linux
netstat -rn

# 或者
route -n

# 或者（更现代）
ip route show
```

**输出示例：**
```
Destination     Gateway         Genmask         Flags   Iface
0.0.0.0         192.168.1.1     0.0.0.0         UG      eth0
192.168.1.0     0.0.0.0         255.255.255.0   U       eth0
```

解读：
- 第一行：去任何地方（0.0.0.0），都先发给网关（192.168.1.1）
- 第二行：去本地网络（192.168.1.x），直接发送

### 🗺️ 每一跳都查表

数据包从你电脑到目标服务器，**每经过一个路由器，都会查一次路由表**：

```
你的电脑 → 家里路由器 → 小区路由器 → ISP路由器 → ... → 目标服务器
         ↓查表        ↓查表         ↓查表
```

### 💻 traceroute：看到每一跳

```bash
# Mac/Linux - 追踪到百度的路径
traceroute www.baidu.com

# Mac上也可以用
traceroute -I www.baidu.com
```

**输出示例：**
```
 1  192.168.1.1 (192.168.1.1)  2.123 ms    # 家里路由器
 2  10.0.0.1 (10.0.0.1)  5.456 ms         # 小区路由器
 3  61.135.169.1 (61.135.169.1)  10.789 ms # ISP路由器
 4  202.106.0.1 (202.106.0.1)  15.234 ms   # 骨干网
 ...
10  220.181.38.148 (220.181.38.148)  25.678 ms  # 百度服务器
```

每一行就是一跳（hop），能看到：
- 经过了哪些路由器
- 每一跳的延迟

⚠️ **注意**：有些路由器不响应traceroute，会显示 `* * *`。

## 5. 协议关系总结：分层协作

网络协议是**分层设计**的，每层负责不同的事：

```
┌─────────────────────────────────────┐
│  应用层：HTTP/HTTPS/SSH/FTP         │  定义"说什么话"
│  （浏览器、邮件客户端）              │  
├─────────────────────────────────────┤
│  传输层：TCP/UDP + 端口号           │  定义"怎么送"
│  （保证可靠 vs 追求速度）            │  
├─────────────────────────────────────┤
│  网络层：IP地址 + 路由              │  定义"送到哪"
│  （找到目标机器）                    │  
├─────────────────────────────────────┤
│  链路层：MAC地址 + 以太网           │  定义"怎么在本地网络传"
│  （同一局域网内传输）                │  
└─────────────────────────────────────┘
```

### 📦 一个HTTP请求的完整过程

当你在浏览器输入 `https://example.com/index.html`：

1. **应用层**：浏览器构造HTTP请求
   ```
   GET /index.html HTTP/1.1
   Host: example.com
   ```

2. **传输层**：选择TCP协议，目标端口443（HTTPS）
   - 先三次握手建立连接
   - 把HTTP请求切成多个TCP段

3. **网络层**：DNS查询得到IP地址（比如93.184.216.34）
   - 加上IP头部（源IP、目标IP）
   - 查路由表，决定发给哪个网关

4. **链路层**：加上MAC地址头部
   - 通过网卡发送到物理网络

5. **中间路由器**：每一跳都重复"查路由表→转发"

6. **目标服务器**：反向解包
   - 链路层去掉MAC头
   - 网络层去掉IP头
   - 传输层去掉TCP头，重组数据
   - 应用层nginx收到HTTP请求，返回网页

## 6. 动手验证：实战命令

### 💻 实验1：用curl发HTTP请求

```bash
# Mac/Linux - 访问网页，查看响应头
curl -I https://www.baidu.com

# 查看完整响应
curl -v https://www.baidu.com

# 指定请求方法
curl -X POST https://api.example.com/data

# 带参数
curl "https://api.example.com/search?q=hello"

# 带请求头
curl -H "Content-Type: application/json" https://api.example.com
```

**输出解读：**
```bash
curl -v https://www.baidu.com

# 输出：
* Connected to www.baidu.com (220.181.38.148) port 443  # 连接到IP和端口
* SSL connection using TLSv1.3                          # 使用HTTPS加密
> GET / HTTP/1.1                                        # 发送HTTP请求
> Host: www.baidu.com
> User-Agent: curl/7.79.1
> 
< HTTP/1.1 200 OK                                       # 收到响应：200成功
< Server: bfe
< Content-Type: text/html
```

### 💻 实验2：查看本机监听的端口

```bash
# Mac/Linux - 查看所有监听端口
sudo lsof -iTCP -sTCP:LISTEN -n -P

# 只看端口号
sudo lsof -iTCP -sTCP:LISTEN -n -P | awk '{print $9}' | sort -u
```

**实验：启动一个简单的Web服务器**
```bash
# Mac/Linux - Python启动HTTP服务器（端口8000）
cd ~
python3 -m http.server 8000

# 另一个终端查看
sudo lsof -i :8000

# 浏览器访问
# http://localhost:8000
```

### 💻 实验3：测试端口连通性

```bash
# Mac/Linux - 测试端口是否开放
nc -zv example.com 80

# 输出：
# Connection to example.com port 80 [tcp/http] succeeded!

# 测试SSH端口
nc -zv example.com 22

# 测试多个端口
nc -zv example.com 20-25
```

### 💻 实验4：抓包看协议

```bash
# Mac/Linux - 抓取HTTP流量（需要sudo）
sudo tcpdump -i any port 80 -A

# 抓取特定主机的流量
sudo tcpdump -i any host 192.168.1.100

# 保存到文件
sudo tcpdump -i any port 80 -w capture.pcap
```

⚠️ **注意**：tcpdump需要管理员权限，抓包文件可以用Wireshark打开分析。

## 本章小结

### 🔑 核心概念

1. **端口号**：区分同一台电脑上的不同服务（0-65535）
   - 常见端口：22(SSH)、80(HTTP)、443(HTTPS)、3306(MySQL)
   - 用 `lsof` 或 `netstat` 查看端口占用

2. **TCP vs UDP**：
   - TCP = 打电话（可靠、慢）→ 网页、文件、邮件
   - UDP = 发短信（快、可能丢）→ 直播、游戏、DNS

3. **应用层协议**：
   - HTTP/HTTPS：网页浏览，状态码表达结果
   - SSH：安全远程登录（端口22）
   - FTP/SFTP：文件传输

4. **路由**：数据包每经过一个路由器，都查路由表决定下一跳
   - 用 `traceroute` 看到完整路径

5. **协议分层**：
   - 应用层定义"说什么"
   - 传输层定义"怎么送"
   - 网络层定义"送到哪"

### 💡 记忆口诀

```
端口号是房间号，区分服务不会乱
TCP可靠像电话，UDP快速像短信
HTTP明文HTTPS密，状态码表达结果
路由表是地图册，每跳查表找出路
```

## 📦 扩展区域

### 🔬 深入话题

#### 1. TCP三次握手和四次挥手

**三次握手（建立连接）：**
```
客户端                    服务器
  │                         │
  │───── SYN ──────────────>│  1. 客户端：我想连接
  │                         │
  │<──── SYN-ACK ───────────│  2. 服务器：可以，你准备好了吗
  │                         │
  │───── ACK ──────────────>│  3. 客户端：准备好了
  │                         │
  │      连接建立             │
```

**四次挥手（断开连接）：**
```
客户端                    服务器
  │                         │
  │───── FIN ──────────────>│  1. 客户端：我说完了（FIN = Finish，结束）
  │                         │
  │<──── ACK ───────────────│  2. 服务器：知道了
  │                         │
  │<──── FIN ───────────────│  3. 服务器：我也说完了
  │                         │
  │───── ACK ──────────────>│  4. 客户端：知道了
  │                         │
  │      连接关闭             │
```

**为什么握手3次，挥手4次？**
- 握手时，服务器的SYN和ACK可以合并发送
- 挥手时，服务器收到FIN后，可能还有数据要发，所以ACK和FIN分开

#### 2. 端口扫描与安全

**常见扫描工具：**
```bash
# nmap扫描端口（需要安装）
nmap -p 1-1000 example.com

# 扫描常见端口
nmap -F example.com

# 扫描并识别服务版本
nmap -sV example.com
```

⚠️ **安全提示**：
- 只扫描自己的服务器，扫描别人的服务器可能违法
- 生产服务器应该关闭不必要的端口
- 使用防火墙限制端口访问

#### 3. HTTP/2 和 HTTP/3

| 版本 | 传输层 | 特点 | 发布时间 |
|------|--------|------|----------|
| HTTP/1.1 | TCP | 一个连接一次只能发一个请求 | 1997 |
| HTTP/2 | TCP | 多路复用，一个连接可以并发多个请求 | 2015 |
| HTTP/3 | UDP(QUIC) | 基于UDP，更快的连接建立 | 2022 |

**查看网站使用的HTTP版本：**
```bash
# Mac/Linux
curl -I --http2 https://www.cloudflare.com

# 输出会显示：
# HTTP/2 200
```

#### 4. WebSocket：持久连接

HTTP是"一问一答"，WebSocket是"持久对话"：

```
HTTP:      客户端问 → 服务器答 → 连接关闭 → 再问再答
WebSocket: 客户端 ←─────持久连接─────→ 服务器
                    双方随时可以发消息
```

**应用场景：**
- 在线聊天（微信网页版）
- 实时协作（腾讯文档）
- 股票行情推送
- 在线游戏

#### 5. 常用端口速记技巧

| 端口 | 服务 | 记忆方法 |
|------|------|----------|
| 22 | SSH | "二二"得四，安全登录 |
| 80 | HTTP | "八零"后，互联网一代 |
| 443 | HTTPS | "四四三"，安全网页 |
| 3306 | MySQL | "33"像两个数据库图标 |
| 5432 | PostgreSQL | "54"是PostgreSQL的缩写PG在键盘上的位置 |
| 6379 | Redis | MERZ在手机键盘上对应6379 |
| 8080 | HTTP备用 | 80的兄弟，开发常用 |
| 27017 | MongoDB | Mongo的默认端口，记住就行 |
