# 第 5 章 · 网络边界与隔离 —— NAT、防火墙、VLAN、DMZ

## 这章解决什么问题？

前面三章我们学会了设备怎么连接、数据怎么传输、怎么找到目标地址。但现在有个新问题：

- **家里只有一个公网IP，但有10台设备要上网，怎么办？**
- **服务器对外开放，怎么防止黑客攻击？**
- **公司财务部门和技术部门在同一个网络，怎么隔离？**
- **为什么我的服务配置好了，但外面就是访问不了？**

这一章就是解决"边界管理"和"内部隔离"的问题。就像一栋大楼需要门禁、保安、不同楼层的门禁卡一样，网络也需要这些安全机制。

## 1. NAT：多设备共享一个公网IP

### 1.1 问题场景

你家里有：手机、电脑、平板、智能电视、游戏机……10台设备。但运营商只给你一个公网IP地址（比如 `123.45.67.89`）。

**怎么让这10台设备同时上网？**

答案就是 **NAT（Network Address Translation，网络地址转换）**。

### 1.2 NAT的核心作用

🔑 **一句话概括**：NAT 就是路由器把**内网地址**换成**公网地址**。

- **出去时**：把数据包的源地址从 `192.168.1.100` 改成 `123.45.67.89`
- **回来时**：把数据包的目标地址从 `123.45.67.89` 改回 `192.168.1.100`

#### 类比：公司前台电话系统

想象一个公司：

- **对外**：只有一个总机号码 `400-123-4567`
- **对内**：每个员工有分机号（8001、8002、8003……）

当员工打电话出去：
1. 员工拨9转外线
2. 对方看到的来电显示是总机号码 `400-123-4567`
3. 对方不知道是哪个分机打的

当外面打电话进来（转接）：
1. 总机接听："您好，请问找哪位？"
2. 转接到对应分机

**NAT就是这个前台系统！** 路由器就是那个"总机"。

### 1.3 NAT表：记录"内网地址 ↔ 公网地址"的对应关系

**NAT表**是路由器用来"记账"的本子，记录了"内网地址和公网地址的对应关系"。

#### NAT表长什么样？

```
┌───────────────────────┬─────────────────────┐
│ 内网地址               │ 对应的公网地址        │
├───────────────────────┼─────────────────────┤
│ 192.168.1.100:54321   │ 123.45.67.89:12345  │
│ 192.168.1.101:49876   │ 123.45.67.89:12346  │
└───────────────────────┴─────────────────────┘
```

🔑 **核心逻辑**：
- 路由器有一个公网IP（`123.45.67.89`）
- 内网每台设备要上网时，路由器分配一个不同的**公网端口**
- 用"公网端口"来区分是哪台内网设备

**为什么用端口区分？**

因为公网IP只有一个，但端口有65535个。路由器用不同的端口来代表不同的内网设备：
- `123.45.67.89:12345` = 电脑
- `123.45.67.89:12346` = 手机
- `123.45.67.89:12347` = 平板

---

### 1.4 NAT的完整工作过程

假设你家里：
- 路由器公网IP：`123.45.67.89`
- 电脑内网IP：`192.168.1.100`
- 手机内网IP：`192.168.1.101`

#### 场景：电脑和手机同时访问百度

**第1步：电脑和手机发出请求**

```
电脑发出的数据包：
  源地址：192.168.1.100:54321
  目标：220.181.38.148:80（百度）

手机发出的数据包：
  源地址：192.168.1.101:49876
  目标：220.181.38.148:80（百度）
```

**第2步：路由器收到请求，分配端口，记录NAT表**

```
路由器：
  电脑要出去 → 分配公网端口12345 → 记入NAT表
  手机要出去 → 分配公网端口12346 → 记入NAT表

NAT表变成：
┌───────────────────────┬─────────────────────┐
│ 内网地址               │ 对应的公网地址        │
├───────────────────────┼─────────────────────┤
│ 192.168.1.100:54321   │ 123.45.67.89:12345  │
│ 192.168.1.101:49876   │ 123.45.67.89:12346  │
└───────────────────────┴─────────────────────┘
```

**第3步：路由器修改数据包的源地址，发出去**

```
电脑的数据包被改成：
  源地址：123.45.67.89:12345  ← 改成路由器的公网IP+端口
  目标：220.181.38.148:80

手机的数据包被改成：
  源地址：123.45.67.89:12346  ← 同样的公网IP，不同的端口
  目标：220.181.38.148:80
```

百度看到的是同一个IP（`123.45.67.89`），但端口不同（12345和12346），所以知道是两个不同的连接。

**第4步：百度回复**

```
回复给电脑的数据：
  目标地址：123.45.67.89:12345

回复给手机的数据：
  目标地址：123.45.67.89:12346
```

**第5步：路由器收到回复，查NAT表，转发给对应的内网设备**

```
路由器收到发往12345的数据：
  查NAT表：12345 → 192.168.1.100:54321
  把目标地址改回内网地址，转发给电脑

路由器收到发往12346的数据：
  查NAT表：12346 → 192.168.1.101:49876
  把目标地址改回内网地址，转发给手机
```

---

### 1.5 NAT的单向性

🔑 **关键特性**：NAT表是"按需建立"的——只有**内网主动访问外网**时，路由器才会在NAT表里创建记录。

- ✅ **内网主动访问外网**：可以（你访问百度，路由器自动创建NAT记录）
- ❌ **外网主动访问内网**：不行（外面的人访问你的公网IP，NAT表里没有记录，路由器不知道该转发给谁）

#### 为什么外面进不来？

```
外面的人 → 访问 123.45.67.89:12345
           ↓
路由器查NAT表 → 没有12345的记录
           ↓
（这个记录只有在你主动访问外网时才会创建）
           ↓
不知道给谁 → 丢弃
```

#### 什么是"内网穿透"？

**内网穿透** = 让外网的人能够访问你内网的服务。

由于NAT的单向性，外网默认访问不了内网。但有时候你需要让外面访问你，比如：
- 在家里搭建了网站，想让朋友访问
- 远程连接家里的NAS
- 公司内网的服务器需要对外开放

这就是"内网穿透"要解决的问题。

#### 内网穿透的几种方法

| 方法 | 原理 | 适用场景 | 章节 |
|------|------|---------|------|
| **端口转发** | 在路由器上手动配置"公网某端口 → 内网某设备" | 有公网IP，路由器可配置 | [第7章](08_home_network.md) |
| **DMZ主机** | 把某台内网设备完全暴露在公网 | 临时测试，不推荐长期使用 | 本章第5节 |
| **内网穿透工具** | 借助一台有公网IP的服务器做中转 | 没有公网IP，或运营商封端口 | [第7章](08_home_network.md) |

---

### 1.6 小结：NAT的核心逻辑

| 动作 | 路由器做了什么 | NAT表的作用 |
|------|--------------|------------|
| 内网→外网（出去） | 把源地址从内网IP改成公网IP | 记录"这个公网端口对应哪个内网设备" |
| 外网→内网（回来） | 把目标地址从公网IP改回内网IP | 查表"这个公网端口的回复该给谁" |

🔑 **记住**：
- NAT表的核心就是两个字：**映射**（内网地址 ↔ 公网地址）
- NAT表的记录是**动态创建**的，只有你主动访问外网时才会有
- 想让外面主动访问你？需要手动配置**端口转发**

## 2. 防火墙：网络的"保安系统"

### 2.1 为什么家里也需要防火墙？

你可能觉得：我电脑上又没开SSH，也没搭网站，别人能访问我什么？

**答案是：你电脑上可能有一些你自己都不知道在运行的服务。**

#### 普通人电脑上可能有的服务

| 系统 | 可能默认开启的服务 | 风险 |
|------|-------------------|------|
| **Windows** | 文件共享（SMB 445端口）、网络发现、远程桌面（RDP 3389端口）、打印机共享 | 勒索病毒常利用SMB传播 |
| **macOS** | AirDrop、文件共享（AFP/SMB）、屏幕共享、AirPlay接收 | 局域网内别人可能看到你的文件 |
| **智能电视/摄像头** | 各种IoT服务 | 被黑客控制当"肉鸡" |

#### 两个真实场景

**场景1：2017年 WannaCry 勒索病毒**
- 利用 Windows 的 SMB 文件共享漏洞（445端口）
- 只要你连着网，没有打补丁，就可能中招
- 中毒后文件被加密，要交比特币才能解密

**场景2：局域网内的"偷窥"**
- 你在咖啡店连WiFi，开了"网络发现"
- 别人可以在"网络邻居"里看到你的电脑
- 如果你共享了文件夹，别人能直接访问

#### 所以防火墙的作用是

1. **挡住你不小心暴露的服务**（比如忘记关的文件共享）
2. **挡住恶意扫描**（黑客扫描整个网段，看谁有漏洞）
3. **增加攻击成本**（即使有漏洞，防火墙拦着也进不来）

### 2.2 防火墙在做什么？

🔑 **一句话**：防火墙就是**按规则决定**哪些数据包能通过，哪些不行。

**类比**：小区门口的保安
- 业主可以自由进出 → 放行
- 外卖员需要登记 → 检查后放行
- 可疑人员直接拦下 → 拒绝

防火墙就是这个"保安"，只不过它检查的是**网络数据包**，不是人。

### 2.3 一个规则长什么样？

防火墙的规则很简单，就是：**"如果符合条件X，就执行动作Y"**

#### 最简单的例子

```
规则1：允许所有人访问80端口
规则2：拒绝其他所有访问
```

这就是一个最简单的防火墙配置。80端口是网站用的，其他端口都不开放。

#### 稍微复杂一点的例子

```
规则1：允许所有人访问80端口（网站）
规则2：允许所有人访问443端口（HTTPS）
规则3：只允许公司IP（203.0.113.0/24）访问22端口（SSH）
规则4：拒绝其他所有入站
```

这个配置的意思是：
- 网站对所有人开放
- SSH（远程登录）只允许公司内部访问
- 其他端口一律关闭

### 2.4 入站 vs 出站

防火墙规则分为两个方向：

| 方向 | 意思 | 例子 |
|------|------|------|
| **入站** | 外面的数据**进来** | 别人访问你的网站 |
| **出站** | 里面的数据**出去** | 你访问百度 |

#### 普通人的电脑怎么配？

```
入站：默认拒绝（我不对外提供服务）
出站：默认允许（我要访问各种网站）
```

#### 服务器怎么配？

```
入站：只开放必要的端口（80、443、22等）
出站：默认允许（服务器需要更新软件、访问数据库等）
```

### 2.5 防火墙在哪里？

防火墙不是一个东西，而是一种**功能**。这个功能可以由不同的设备/软件来实现：

| 位置 | 例子 | 你需要关心吗？ |
|------|------|--------------|
| **家用路由器** | 路由器自带的防火墙 | 通常不用管，默认配置就够用 |
| **电脑系统** | Windows防火墙、macOS防火墙 | 建议开启，用默认配置 |
| **云服务器** | 阿里云/腾讯云的"安全组" | **需要手动配置！** |

🔑 **关键点**：如果你买了云服务器，安全组就是你的第一道防火墙，必须在云控制台配置。

### 2.6 云服务器怎么配置安全组？

假设你买了一个阿里云服务器，想搭一个网站：

**第1步**：登录阿里云控制台 → 安全组 → 添加规则

| 方向 | 端口 | 源地址 | 动作 |
|------|------|--------|------|
| 入站 | 22 | 0.0.0.0/0 | 允许（SSH登录） |
| 入站 | 80 | 0.0.0.0/0 | 允许（HTTP网站） |
| 入站 | 443 | 0.0.0.0/0 | 允许（HTTPS网站） |
| 入站 | 其他 | - | 拒绝（默认） |

**第2步**：在服务器内部，也要确保防火墙开了这些端口（或者直接关闭服务器防火墙，只依赖云安全组）

### 2.7 常见问题：服务配好了但访问不了

⚠️ **90%的"服务配好但访问不了"问题都是防火墙导致的！**

#### 排查顺序

```
1. 服务在运行吗？
   └─ 在服务器上执行：curl http://localhost:80
   └─ 能访问 → 服务没问题，是防火墙的问题
   └─ 不能访问 → 服务本身有问题

2. 云安全组开了吗？（云服务器）
   └─ 登录云控制台检查

3. 服务器防火墙开了吗？
   └─ Linux: sudo ufw status
   └─ 看看80端口在不在允许列表

4. 路由器端口转发配了吗？（家里服务器）
   └─ 登录路由器检查
```

🔑 **记住**：防火墙是**层层把关**的。云安全组开了，服务器防火墙没开，照样访问不了。

## 3. VLAN：逻辑隔离

### 3.1 什么是VLAN？

**VLAN（Virtual Local Area Network，虚拟局域网）**：在同一个物理网络上，划分出多个逻辑上隔离的网络。

🔑 **类比**：写字楼里的不同公司

一栋写字楼有100家公司：
- 物理上：大家都在同一栋楼，用同一套电梯、走廊
- 逻辑上：每家公司有自己的门禁卡，只能进自己的办公区

**VLAN就是这个门禁系统！**

### 3.2 为什么需要VLAN？

#### 场景：公司网络

一家公司有：
- 财务部门（10台电脑）
- 技术部门（20台电脑）
- 访客WiFi（不限数量）

**问题**：
- 如果都在同一个局域网，技术部门可以访问财务部门的电脑（不安全）
- 访客WiFi可以访问公司内网（更不安全）

**解决方案**：用VLAN隔离

```
VLAN 10：财务部门（192.168.10.0/24）
VLAN 20：技术部门（192.168.20.0/24）
VLAN 30：访客WiFi（192.168.30.0/24）
```

- VLAN 10和VLAN 20之间默认不能互相访问
- VLAN 30完全隔离，只能访问外网

### 3.3 VLAN的实现

VLAN需要支持的交换机（Managed Switch）。

**配置示例**（思科交换机）：

```
# 创建VLAN
Switch(config)# vlan 10
Switch(config-vlan)# name Finance

Switch(config)# vlan 20
Switch(config-vlan)# name Tech

# 把端口分配到VLAN
Switch(config)# interface FastEthernet 0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10

Switch(config)# interface FastEthernet 0/2
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 20
```

### 3.4 家庭需要VLAN吗？

🔑 **答案**：普通家庭不需要。

- 家里设备少，没有隔离需求
- 家用路由器通常不支持VLAN
- 如果真的需要隔离，可以用"访客WiFi"功能（本质上是简化版的VLAN）

**适用场景**：
- 中大型公司
- 学校、医院等机构
- 智能家居发烧友（隔离IoT设备）

## 4. 网段规划：方便管理

### 4.1 为什么要规划网段？

如果公司有100台设备，都在 `192.168.1.0/24` 网段，你怎么知道哪台是财务的、哪台是技术的？

**解决方案**：不同部门用不同网段。

```
192.168.10.0/24 → 财务部门
192.168.20.0/24 → 技术部门
192.168.30.0/24 → 市场部门
192.168.99.0/24 → 服务器
192.168.100.0/24 → 访客WiFi
```

### 4.2 规划原则

1. **预留空间**：每个网段不要用满，留一些IP备用
2. **有意义的编号**：比如10、20、30，而不是11、17、23
3. **服务器单独网段**：方便配置防火墙规则
4. **文档记录**：一定要写文档，不然过几个月自己都忘了

### 4.3 示例：小公司网络规划

```
网段                  用途              设备数量
192.168.1.0/24      管理网络（路由器等）  10
192.168.10.0/24     财务部门            20
192.168.20.0/24     技术部门            50
192.168.30.0/24     市场部门            30
192.168.99.0/24     服务器              10
192.168.100.0/24    访客WiFi            不限
```

## 5. DMZ：对外服务区域

### 5.1 什么是DMZ？

**DMZ（Demilitarized Zone，非军事区）**：专门用来放置对外服务的区域。

🔑 **类比**：酒店的大堂

- **大堂**：任何人都可以进（DMZ，对外开放）
- **客房区**：只有客人能进（内网，受保护）
- **员工区**：只有员工能进（核心内网，更严格）

### 5.2 为什么需要DMZ？

#### 场景：公司有一台Web服务器

**问题**：
- Web服务器要对外开放80端口（任何人都能访问）
- 但公司内网有财务数据、客户信息（不能被外面访问）

**如果Web服务器和内网在一起**：
- 万一Web服务器被黑客攻破，黑客就能访问整个内网

**解决方案**：把Web服务器放在DMZ

```
互联网
  ↓
防火墙1（只允许80/443端口）
  ↓
DMZ区域（Web服务器）
  ↓
防火墙2（严格限制）
  ↓
内网（财务、客户数据）
```

### 5.3 家用路由器的DMZ功能

家用路由器的"DMZ主机"功能：把某台设备完全暴露在公网。

⚠️ **注意**：这不是真正的DMZ，只是"把所有端口都转发给这台设备"。

**使用场景**：
- 玩游戏需要开放所有端口（但不推荐，不安全）
- 测试服务器（临时用）

**配置方法**：
1. 登录路由器管理界面（通常是 `192.168.1.1`）
2. 找到"DMZ主机"或"DMZ Host"
3. 输入要暴露的设备IP（比如 `192.168.1.100`）
4. 保存

⚠️ **安全提醒**：DMZ主机会完全暴露在公网，务必做好防火墙配置！

## 6. "转发"和"代理"到底是什么意思？

网络里有很多词：路由转发、NAT转发、端口转发、代理、反向代理……这些词很容易混淆。这一节一次性说清楚。

### 6.1 "转发"是什么意思？

**转发 = 收到数据后，替你送到另一个地方。**

就像"传话"——你把话告诉中间人，中间人替你传给目标。

但"传话"的方式不同，就产生了不同类型的转发：

| 类型 | 谁在转发 | 干了什么 | 生活类比 |
|------|---------|---------|---------|
| **路由转发** | 路由器 | 看地址，把数据送到下一站 | 邮局分拣员：看地址，扔到对应的邮袋 |
| **NAT转发** | 路由器 | 把你的内网IP换成公网IP再发出去 | 公司前台：员工打外线，对方看到的是总机号 |
| **端口转发** | 路由器 | 把公网某端口的流量转到内网某设备 | 前台接电话：外面打来找8001分机，转给对应员工 |

### 6.2 "代理"是什么意思？

**代理 = 中间人帮你办事。**

你 → 代理 → 目标

代理代替你去和目标打交道，目标可能不知道真正的你是谁。

### 6.3 正向代理 vs 反向代理

关键区别：**代理在帮谁？**

| | 正向代理 | 反向代理 |
|--|---------|---------|
| **帮谁** | 帮客户端（你） | 帮服务器 |
| **藏谁** | 藏着你（服务器不知道你是谁） | 藏着服务器（你不知道真正的服务器是谁） |
| **例子** | 朋友帮你打电话 | 400客服电话 |

#### 正向代理：朋友帮你打电话

```
你想给某人打电话，但不想让他知道是你
    ↓
你让朋友帮你打
    ↓
对方只看到朋友的来电显示
    ↓
对方不知道是你想找他
```

**生活中的正向代理：**
- 翻墙工具（VPN）——你访问Google，Google只看到代理服务器的IP
- 公司上网代理——你访问网站，网站只看到公司的IP

#### 反向代理：400客服电话

```
你打 400-xxx-xxxx 客服电话
    ↓
系统自动分配给某个客服接听
    ↓
你不知道是哪个客服接的
    ↓
你只知道打了400，具体谁服务你不管
```

**生活中的反向代理：**
- 百度/淘宝——你访问 baidu.com，其实后面有成千上万台服务器，你不知道具体访问的是哪台
- Nginx 负载均衡——把请求分发给不同的后端服务器

### 6.4 反向代理有什么用？

**用途1：一台机器跑多个服务**

你的服务器上有：
- 网站（端口 8080）
- API服务（端口 3000）

**没有反向代理：**
```
用户要访问：http://1.2.3.4:8080  （网站，要记端口号）
用户要访问：http://1.2.3.4:3000  （API，要记端口号）
```

**有反向代理（Nginx）：**
```
用户访问：http://1.2.3.4/       → Nginx 转发给 8080
用户访问：http://1.2.3.4/api/   → Nginx 转发给 3000
```

用户不需要记端口号，只需要访问同一个地址。

**用途2：多台服务器分担流量（负载均衡）**

你的网站火了，一台服务器扛不住，买了10台。

```
用户访问：http://your-site.com
           ↓
      Nginx（反向代理）
           ↓
    挑一台有空的服务器
           ↓
    ┌───┬───┬───┐
    ↓   ↓   ↓   ↓
   服务器1 服务器2 服务器3 ...
```

用户只访问一个地址，Nginx 自动分配到不同服务器。

### 6.5 怎么配置反向代理？（Nginx）

**步骤1：安装 Nginx**

```bash
# Ubuntu/Debian
sudo apt install nginx
```

**步骤2：编辑配置文件**

`/etc/nginx/sites-available/default`：

```nginx
server {
    listen 80;
    server_name your-server.com;

    # 访问 / 转发给 8080 端口
    location / {
        proxy_pass http://localhost:8080;
    }

    # 访问 /api/ 转发给 3000 端口
    location /api/ {
        proxy_pass http://localhost:3000/;
    }
}
```

**步骤3：重启 Nginx**

```bash
sudo nginx -t              # 检查配置
sudo systemctl restart nginx
```

**核心配置就这一行：**
```nginx
proxy_pass http://localhost:端口;
```

### 6.6 SSH 隧道：一种特殊的代理

SSH 隧道也是一种代理，只是它是加密的、临时的。

**场景：你在咖啡店，想访问公司内网的数据库**

```
你（咖啡店）
    ↓
    直接访问 ✗ 被拒绝（数据库只在公司内网）

但是，你可以 SSH 登录公司的跳板机
```

**SSH 隧道的做法：**

```bash
ssh -L 3306:192.168.1.50:3306 user@jump.company.com
        ↑         ↑
        │         └── 目标：内网数据库的地址和端口
        │
        └── 本地：在你电脑上开的端口
```

意思：**在你电脑上开一个 3306 端口，所有发到这个端口的数据，通过 SSH 转发给内网数据库。**

**效果：**

```
你的电脑                    跳板机                   内网数据库
MySQL客户端 → localhost:3306
              ↓
         SSH隧道（加密）
              ↓
         发给跳板机 ──────────→ 收到，解密
                                      ↓
                                 转发给数据库 ───→ 处理请求
                                      ↓
                              收到数据库回复
                                      ↓
         ←───────────────── 通过 SSH 隧道发回
         ↓
    你的MySQL客户端收到结果
```

### 6.7 总结：各种"代理/转发"一览

| 名称 | 谁在帮谁 | 用途 |
|------|---------|------|
| **正向代理** | 帮客户端 | 翻墙、隐藏身份 |
| **反向代理** | 帮服务器 | 负载均衡、统一入口 |
| **SSH隧道** | 帮客户端（加密） | 访问内网服务 |
| **端口转发** | 帮服务器（在路由器上配） | 让外网能访问内网服务 |

**一句话记住：**
- 正向代理 = 朋友帮你打电话（藏着你）
- 反向代理 = 400客服电话（藏着客服）

### 6.9（旧）更多代理/VPN 内容

#### 先说本质区别

| | VPN | 代理 |
|---|---|---|
| **本质** | 你"加入"了VPS的网络 | 你还在原地，VPS帮你转发 |
| **你的IP** | 变成VPN网络的IP（如10.8.0.2） | 不变，还是原来的 |
| **网络身份** | 你"搬"到了VPS那边 | 你还在原地，找VPS帮忙 |

**一句话：VPN让你"搬家"，代理让你"找人帮忙"。**

---

#### 两者都需要VPS

**VPN和代理都需要一台在境外的服务器（VPS）。**

```
┌─────────────────────────────────────────────────────┐
│                    VPS（境外服务器）                  │
├─────────────────────────────────────────────────────┤
│   装 WireGuard / OpenVPN  →  VPN服务器              │
│   装 Shadowsocks / V2Ray   →  代理服务器             │
│                                                     │
│   VPS是中性的，装什么软件就变成什么服务器             │
└─────────────────────────────────────────────────────┘
```

---

#### VPN的工作原理

**VPN的核心：让你"加入"VPS的网络。**

```
连接VPN后发生了什么：

1. 你的电脑和VPS建立加密隧道
2. 你的电脑获得VPN网络的IP（如10.8.0.2）
3. 系统路由表被修改，所有流量走VPN
4. 结果：你就像真的在VPS那个网络里一样

         你的电脑              VPS              目标网站
         ┌─────────┐        ┌─────────┐        ┌─────────┐
         │ 新IP:   │        │         │        │         │
         │10.8.0.2 │ ────── │ VPN     │ ────── │ 看到VPS │
         │         │  隧道  │ 服务器  │        │ 的IP    │
         └─────────┘        └─────────┘        └─────────┘

关键：你的IP变了，你"加入"了VPN网络
```

---

#### 代理的工作原理

**代理的核心：VPS帮你转发请求。**

```
使用代理后发生了什么：

1. VPS上运行代理服务器（监听某端口）
2. 你的电脑上运行代理客户端
3. 浏览器把请求发给代理客户端
4. 代理客户端加密请求，发给VPS
5. VPS解密，访问目标，返回结果

         你的电脑              VPS              目标网站
         ┌─────────┐        ┌─────────┐        ┌─────────┐
         │ 原IP:   │        │ 代理    │        │         │
         │192.168. │ ────── │ 服务器  │ ────── │ 看到VPS │
         │ 1.100   │  加密  │         │        │ 的IP    │
         └─────────┘        └─────────┘        └─────────┘

关键：你的IP不变，你还在原地
```

---

#### 那代理的TUN模式是什么？

现代代理工具（如Clash）有TUN模式，也创建虚拟网卡，看起来和VPN一样。

**但本质还是代理！**

| | VPN | 代理TUN模式 |
|---|---|---|
| **虚拟网卡** | 有，用来获得新IP | 有，用来捕获流量 |
| **你的IP** | 变了 | 没变 |
| **本质** | 你"加入"了网络 | 你还在原地，只是所有流量都委托给代理 |

**打个比方：**
- VPN = 你真的搬到了美国
- 代理TUN = 你还住在中国，但雇了个人帮你处理**所有**事情

**虚拟网卡只是技术手段，不是本质区别。本质区别是你的IP变没变。**

---

#### 用比喻理解

**VPN = 你人飞到了美国**
- 你"搬"到了美国（网络意义上）
- 所有事情都是从美国发出的
- 不用管哪个应用，全部都是美国的身份

**代理 = 让美国的朋友帮你办事**
- 你还在中国，没有搬家
- 你让美国的朋友（VPS）帮你办某些事
- 只有你委托的事情才经过朋友
- 没委托的事情还是你自己直接做

---

#### 核心区别总结

| | VPN | 代理 |
|---|---|---|
| **本质** | 你"加入"VPS的网络 | VPS帮你转发 |
| **你的IP** | 变成VPN网络的IP | 不变 |
| **覆盖范围** | 所有应用自动走VPN | 只有配置的应用才走代理 |
| **工作层次** | 网络层（第3层） | 应用层（第7层） |
| **配置复杂度** | 连接后全自动 | 每个应用可能需要单独配置 |
| **适用场景** | 想简单，连上就不管 | 精确控制哪些应用走代理 |

### 6.9 访问境外网络的原理

#### 为什么有的网站访问不了？

中国有"网络防火墙"（俗称"墙"），会拦截对某些境外网站的访问。

**但墙不是封锁所有境外IP，而是选择性封锁：**

```
墙的黑名单：
├── Google 的 IP 段 → 封
├── YouTube 的 IP 段 → 封
├── Facebook 的 IP 段 → 封
├── Twitter 的 IP 段 → 封
├── 某些 VPS 的 IP → 可能封（如果被发现大量翻墙）
└── 其他大多数 IP → 不封
```

**如果封所有境外IP，中国就和世界断网了，这不可能。**

#### 为什么境外 VPS 能访问？

```
你在中国，想访问两个境外服务器：

情况A：直接访问 Google
┌─────┐         ┌─────┐         ┌─────────┐
│ 你  │ ───×────│ 墙  │         │ Google  │
└─────┘  被拦截 └─────┘         └─────────┘

墙检查：Google 的 IP 在黑名单 → 拦截！


情况B：直接访问你的 VPS
┌─────┐         ┌─────┐         ┌─────────┐
│ 你  │ ────────│ 墙  │ ────────│   VPS   │
└─────┘  放行   └─────┘  放行   └─────────┘

墙检查：这个 VPS 的 IP 不在黑名单 → 放行！
```

**所以关键点是：VPS 的 IP 没被封 + Google 的 IP 被封 = 你能连 VPS，VPS 能连 Google**

#### 完整的访问流程

```
你                     VPS                    Google
│                       │                       │
│  1. 我能连你吗？      │                       │
│ ───────────────────→ │                       │
│                       │                       │
│  2. 能，我的IP没被封  │                       │
│ ←─────────────────── │                       │
│                       │                       │
│  3. 帮我访问Google    │                       │
│ ───────────────────→ │                       │
│                       │  4. 访问Google        │
│                       │ ────────────────────→│
│                       │                       │
│                       │  5. 返回Google页面    │
│                       │ ←────────────────────│
│  6. 这是Google的页面  │                       │
│ ←─────────────────── │                       │

墙只看到第1-2步（你和VPS通信），看不到第4-5步（VPS和Google通信）
而且第1-3步是加密的，墙不知道你在访问什么
```

#### 需要什么？

**两样东西：**

1. **一台境外的服务器（VPS）** — 在那边帮你转发流量
2. **VPN或代理软件** — 让你的电脑和VPS建立加密连接

#### 怎么搭建？

**方法1：自己买VPS搭建**

```
1. 买一台境外VPS（Vultr、DigitalOcean等，月费$3-10）
2. 在VPS上安装服务端软件：
   - VPN方案：WireGuard、OpenVPN
   - 代理方案：Shadowsocks、V2Ray、Xray
3. 在你电脑上安装客户端
4. 连接VPS，开始使用
```

**方法2：用现成的服务**

```
直接买VPN服务或"机场"服务：
- 人家帮你准备好了VPS和软件
- 你只需要安装客户端，填入配置
- 省事，但隐私和稳定性取决于服务商
```

#### VPS会被封吗？

**会！** 如果很多人用同一个VPS：

```
墙发现这个IP有大量异常流量 → 把IP加入黑名单 → VPS废了
```

- 用的人少的VPS → 不容易被封
- 很多人用的公共VPN → 容易被封

#### 更高级的伪装

现代代理工具不只是靠"IP不在黑名单"，还会**伪装流量**：

```
普通的代理流量：
你 ──代理协议数据──→ VPS
墙一看："这是代理协议，封！"

伪装后的流量（V2Ray/Trojan）：
你 ──看起来像HTTPS──→ VPS
墙一看："这是普通 HTTPS 访问网站，放行"
```

V2Ray、Trojan 这些工具就是把流量伪装成普通的 HTTPS，让墙看不出是代理。

### 6.10 一句话总结

```
VPS = 境外服务器（VPN和代理都需要它）

VPN = 你"搬"到VPS那边（IP变了，加入网络）
代理 = 你还在原地，让VPS帮你办事（IP不变，转发请求）

本质区别：你的IP变没变
```

## 7. 概念关系：NAT、防火墙、VLAN

很多人容易混淆这三个概念，我们来理清关系：

### 7.1 功能对比

| 技术 | 主要功能 | 类比 | 典型场景 |
|------|---------|------|---------|
| **NAT** | 地址转换，让内网设备共享公网IP | 公司总机 | 家庭路由器 |
| **防火墙** | 过滤数据包，控制进出 | 小区保安 | 服务器安全 |
| **VLAN** | 逻辑隔离，划分虚拟网络 | 楼层门禁 | 公司部门隔离 |

### 7.2 工作层次

```
外网 ←→ [NAT] ←→ [防火墙] ←→ [VLAN] ←→ 内网设备
        管出去      管进来      管内部隔离
```

### 7.3 典型组合

#### 家庭网络
```
互联网 → 路由器（NAT + 简单防火墙） → 所有设备
```

#### 小公司网络
```
互联网 → 路由器（NAT） → 防火墙 → 交换机（VLAN） → 各部门
```

#### 大公司网络
```
互联网
  ↓
边界路由器（NAT）
  ↓
防火墙1（外部防火墙）
  ↓
DMZ区域（Web服务器、邮件服务器）
  ↓
防火墙2（内部防火墙）
  ↓
核心交换机（VLAN）
  ↓
各部门交换机 → 员工电脑
```

## 本章小结

### 核心概念

1. **NAT（网络地址转换）**
   - 让多台设备共享一个公网IP
   - 单向性：内网可以主动访问外网，外网不能主动访问内网
   - 这是"内网穿透"问题的根源

2. **防火墙**
   - 根据规则过滤数据包
   - 入站规则（外面进来）和出站规则（里面出去）
   - Linux工具：ufw（推荐）、iptables（高级）
   - ⚠️ 90%的"服务配好但访问不了"问题都是防火墙导致的

3. **VLAN（虚拟局域网）**
   - 在同一个物理网络上划分多个逻辑网络
   - 用于部门隔离、安全控制
   - 家庭通常不需要

4. **网段规划**
   - 不同部门用不同IP段
   - 方便管理和配置防火墙规则

5. **DMZ（非军事区）**
   - 专门放置对外服务的区域
   - 隔离外部威胁和内网

### 关键要点

- NAT管"出去"，防火墙管"进来"，VLAN管"内部隔离"
- 最小权限原则：默认拒绝，只开放必要的端口
- 云服务器有两层防火墙：云安全组 + 服务器内部防火墙
- 排查网络问题时，先检查服务是否运行，再检查防火墙

### 实用命令速查

```bash
# ufw防火墙（Linux）
sudo ufw enable              # 启用防火墙
sudo ufw status              # 查看状态
sudo ufw allow 80            # 允许80端口
sudo ufw delete allow 80     # 删除规则

# 检查端口监听（Mac和Linux）
sudo netstat -tlnp | grep :80
sudo ss -tlnp | grep :80
lsof -i :80

# 测试端口连通性（Mac和Linux）
telnet 192.168.1.100 80
nc -zv 192.168.1.100 80
```

## 📦 扩展区域

### 1. NAT的类型

- **Full Cone NAT**：最宽松，任何外部主机都可以通过映射的端口访问内网
- **Restricted Cone NAT**：只有内网主机访问过的外部IP才能回来
- **Port Restricted Cone NAT**：不仅限制IP，还限制端口
- **Symmetric NAT**：最严格，每个目标都分配不同的端口

**影响**：P2P应用（如视频通话）在Symmetric NAT下很难建立连接。

### 2. 防火墙的类型

- **包过滤防火墙**：只看IP和端口（iptables的基本功能）
- **状态检测防火墙**：记住连接状态，更智能（iptables的 `-m state`）
- **应用层防火墙**：能识别HTTP、FTP等协议内容（如WAF）
- **下一代防火墙（NGFW）**：集成入侵检测、病毒扫描等（企业级）

### 3. 内网穿透原理

既然NAT导致外网访问不了内网，内网穿透是怎么做到的？

**原理**：借助一台有公网IP的服务器做中转。

```
你的电脑（内网） → 主动连接 → 中转服务器（公网）
外部用户 → 访问中转服务器 → 转发给你的电脑
```

**常用工具**：
- **frp**：开源，需要自己有公网服务器
- **ngrok**：商业服务，免费版有限制
- **ZeroTier**：虚拟局域网，P2P连接

### 4. IPv6与NAT

IPv6地址数量巨大（2^128个），理论上每台设备都可以有公网IP，不需要NAT。

**但现实是**：
- 运营商出于安全考虑，仍然会做防火墙
- 很多设备和网站还不支持IPv6
- NAT在短期内不会消失

### 5. 云原生的网络隔离

在Kubernetes等容器平台，网络隔离用的是：

- **Network Policy**：类似防火墙规则，控制Pod之间的通信
- **Service Mesh**：更高级的流量管理（如Istio）

**示例**（Kubernetes Network Policy）：
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

### 6. 零信任网络（Zero Trust）

传统安全模型：内网是可信的，外网是不可信的。

**零信任模型**：不管在内网还是外网，都不信任，每次访问都要验证身份。

**核心原则**：
- 永远不信任，永远要验证
- 最小权限访问
- 假设网络已被攻破

**实现技术**：
- 微隔离（Micro-segmentation）
- 软件定义边界（SDP）
- 身份认证（如OAuth、SAML）
